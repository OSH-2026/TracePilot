# TracePilot 调研扩展（系统行为 + 高频场景 + 关键线程识别）

作者：贺小轩
---
本文档基于 TracePilot 既有工作，围绕系统行为观测、高频使用场景扩展，以及关键/非关键线程与重载/非重载线程识别方法开展调研性分析。行文以调研报告体例组织，侧重方法归纳、证据链梳理与实践路径总结，不涉及具体实施步骤。

## 0. 研究基础与对齐范围

- 已完成场景：
  - 信息流滚动（Chrome）已有 eBPF 数据与分析报告。
  - QQ 页面切换已形成行为特征分析报告。
  - 已有场景覆盖“连续交互”和“页面跳转”两类负载，便于建立基线与对照。
- 已有指标：
  - `sched_switch`、`sched_waking`、`sched_wakeup`、`cpu_frequency`。
  - 线程级汇总：on-CPU 时间、迁核次数、wakeup-to-run、runnable delay。
  - 指标体系以调度事件为主，尚缺少外部扰动（网络、传感器）解释维度。
- 项目定位：
  - 重点在数据观测与处理，决策与执行不作为重点。
  - 需要扩展系统行为观测与高频场景采集，并增加关键/非关键、重载/非重载的区分方法。
  - 研究重点在于“可解释的观测指标”和“跨场景的稳定判定规则”。

## 1. 系统行为观测扩展（网络、传感器）

### 1.1 网络行为

**研究目标**：在用户行为场景中补充系统侧网络负载的解释维度，用以区分调度瓶颈与网络突发对卡顿的影响，形成“网络扰动—调度异常—体验波动”的因果链条。

**参考依据**：
- Android eBPF 流量监控（官方）支持按 UID 统计流量，并区分前台/后台数据。
  - https://source.android.com/docs/core/data/ebpf-traffic-monitor?hl=zh-cn
  - https://source.android.com/docs/core/architecture/kernel/bpf

**调研结论与实践路径**：
- 以“按 UID 聚合”为核心的流量侧统计逻辑，与现有 eBPF 调度事件按同一时间窗对齐。
- 前台/后台网络分离用于解释卡顿窗口的外部扰动。
- 用流量突发窗与调度突发窗叠加，区分“调度压力”与“网络爆发”造成的性能异常。
 - 建议引入“速率变化”与“包量变化”并行观察，避免仅以总字节量误判小包风暴。
 - 对网络突发窗口进行前后对比，可识别“先网络、后调度”的反应时序特征。
 - 在场景分析中保留“前台应用 UID 与系统服务 UID”的区分，以避免误将系统后台流量归因于前台体验。

**建议指标字段（秒级窗口）**：
- `net_bytes_total`
- `net_bytes_fg`
- `net_bytes_bg`
- `net_packets_total`

**字段来源说明**：
- 上述字段为“统计口径定义”，多为按 UID 与时间窗聚合后的派生字段，并非系统现成同名字段。
- 原始数据通常来自内核或系统侧网络统计计数（如按 UID 的流量统计），经汇总后得到窗口级指标。
- 若仅能获得更粗粒度统计，则应在报告中标注“统计口径差异”，避免与“精细包级”概念混用。

**输出形态（建议）**：
- 形成“调度事件 + 网络行为”的联合窗口特征表。
- 用流量突发窗作为系统扰动标签，辅助解释调度延迟或帧时间异常。
 - 形成“网络扰动强度分层标签”，用于跨场景对比与异常解释。

**可行性与权限评估**：
- 可行性：中-高。网络统计与调度事件同窗对齐在方法层面可行，但数据获取依赖系统侧能力。
- 权限：中等偏高。若仅使用系统已有统计（如按 UID 聚合）可降低权限压力；细粒度包级行为通常需要系统级权限或内核侧支持。
- 贴合主题：高。网络负载是解释调度异常与体验波动的重要外部扰动因素。

### 1.2 传感器行为

**研究目标**：判断用户是否在持续交互及其运动/方向变化，降低后台噪声被误识别为用户行为的风险，并提升“用户交互—系统负载”关联分析的可信度。

**参考依据**：
- Android 传感器框架与采样速率限制
  - https://developer.android.com/guide/topics/sensors/sensors_overview

**调研结论与实践路径**：
- 以加速度计、重力传感器、旋转矢量为主，构造“运动强度标签”。
- 传感器事件聚合为时间窗统计量，用于判断用户是否在持续交互。
- 传感器状态作为“用户行为可信度”标签，避免把后台噪声误当作用户行为。
 - 建议将“静止阈值”与“方向变化阈值”并行设置，降低短时抖动导致的误判。
 - 传感器变化率可用于区分“连续滑动”与“短促点击”两类交互形态。

**建议指标字段**：
- `sensor_motion_level`（低/中/高）
- `sensor_rotation_change`（0/1）
- `sensor_sampling_rate`（固定 UI/游戏档位）

**字段来源说明**：
- 上述字段为“分析型标签/统计口径”，为多传感器采样后计算得到的派生字段，并非系统固有字段。
- 典型来源为传感器事件流的统计与分箱结果，例如幅值、变化率或方向变化计数。
- 若实际采集仅覆盖部分传感器，应在报告中注明“传感器覆盖范围”，以保持口径一致。

**输出形态（建议）**：
- 形成“调度事件 + 传感器状态”的联合特征表。
- 使用传感器强度区分“真实交互窗”和“系统噪声窗”。
 - 对“交互可信度低”的窗口设置降权规则，避免其主导整体结论。

**可行性与权限评估**：
- 可行性：高。传感器数据获取在常规应用权限范围内具备可行性，且与时间窗对齐容易。
- 权限：低-中。多数基础传感器无需高权限，但采样频率与后台采集可能受系统限制。
- 贴合主题：中-高。用于提升用户行为识别可信度，与调度与体验分析形成支撑关系。

### 1.3 预期输出

在原有窗口级表上追加系统字段，形成“用户行为 + 系统行为”联合特征：
- 事件字段：调度 + Binder + 频率
- 系统字段：网络流量 + 传感器状态
 - 目标用于支撑“因果链解释”和“跨场景可比性”分析。

**可行性与权限评估**：
- 可行性：中。联合特征需要跨来源数据对齐，方法上可行，但取决于各来源数据可得性。
- 权限：中。调度事件与网络统计可能涉及更高权限，传感器相对低权限。
- 贴合主题：高。该输出是“系统行为 + 用户行为”联合分析的核心载体。

## 2. 高频场景扩展（支付、视频、游戏）

### 2.1 支付场景（微信/支付宝）

**场景价值（调研结论）**：
- 具有短时高强度交互与敏感体验阈值。
- Binder 调用链多、系统服务依赖明显，适合用于验证线程关键性判定。
 - 流程高度可重复，适合作为“关键链路稳定性”研究的对照场景。

**关注指标**：
- 首屏/页面加载时间
- Binder latency P95
- runnable delay P95 / wakeup-to-run P95
- 网络流量突发
 - 关键线程 on-CPU 占比与迁核次数

**调研结论与实践路径**：
- 将支付场景作为“关键线程识别”的强信号场景，重点观察 UI 线程与 Binder 链路线程。
- 比较支付流程不同阶段的调度延迟变化，区分“冷启动链路”与“前台渲染链路”。
 - 关注支付页面与确认页面的差异化负载，以识别“系统服务依赖强度”变化。

**可行性与权限评估**：
- 可行性：中。场景具备可重复性，但流程受应用安全策略与设备环境影响。
- 权限：中-高。若需要系统级调度与 Binder 观测，权限要求较高；用户态仅能获得部分指标。
- 贴合主题：高。支付流程关键链路清晰，有助于关键线程与调度瓶颈的识别。

### 2.2 视频场景（抖音/B站）

**场景价值（调研结论）**：
- 长时间连续负载，适合识别“调度稳定性”和“网络波动”之间的关系。
- 解码与渲染线程负载明显，适合做关键线程与重载判断。
 - 具备“启动期—稳定期—切换期”的可分段特征，便于阶段性对比。

**关注指标**：
- frame time P95/P99
- jank rate
- on-CPU share
- cpu migration count
- 网络流量波动
 - 解码线程与渲染线程的调度延迟差异

**调研结论与实践路径**：
- 以帧时间与迁核次数的联合变化作为“调度不稳定”的信号。
- 将网络突发与 frame time 同步对齐，区分“渲染侧瓶颈”和“拉流侧瓶颈”。
 - 启动期关注“首帧时间与调度突发”的耦合关系，稳定期关注“抖动频率”。

**可行性与权限评估**：
- 可行性：高。场景持续时间长、负载稳定，适合观察调度与网络联合效应。
- 权限：中。帧时间与调度指标可能来自系统/框架级接口，需评估可访问性。
- 贴合主题：高。视频场景对调度稳定性与系统扰动敏感，是典型高频场景。

### 2.3 游戏场景（轻量游戏或 3D 应用）

**场景价值（调研结论）**：
- 高频输入 + 持续渲染，适合观察高帧率场景中的调度边界。
- GPU 与 CPU 负载协同明显，适合作为极端场景对照。
 - 更容易暴露热控、频率调节导致的性能波动。

**关注指标**：
- frame time P95/P99
- runnable delay / wakeup-to-run
- CPU 频率变化
- PSI（系统压力）
 - 高频输入窗口内的事件密度变化

**调研结论与实践路径**：
- 以 PSI 作为系统重载标签，结合帧时间判断“重载下的关键线程稳定性”。
- 关注 CPU 频率变化与帧时间尖峰的同步性。
 - 将输入密度与帧时间对齐，区分“输入压力”与“渲染瓶颈”。

**可行性与权限评估**：
- 可行性：中-高。场景负载稳定且可放大瓶颈，但对设备性能与热控敏感。
- 权限：中-高。PSI 与频率数据多为系统级指标，用户态可用性受限。
- 贴合主题：高。高帧率与高输入密度有利于揭示调度边界与重载行为。

## 3. 关键/非关键线程与重载/非重载识别

### 3.1 关键线程识别方法

**方法 A：调度强度与低延迟组合**
- 线程 on-CPU 时间高、wakeup-to-run 低、runnable delay 低。
- 适合使用已有 `feed_scroll_threads_summary.csv`。
 - 该方法强调“高产出、低等待”的线程特征，适合识别 UI 线程与关键服务线程。
 - 关键判据可理解为“持续占用 + 响应敏捷”的组合，能排除大量短时噪声线程。
 - 对于长期常驻服务线程，应结合 on-CPU 占比与延迟分位数共同判断，避免单一阈值误判。
 - 若 on-CPU 高但延迟亦高，可能反映“被迫抢占”而非关键链路，应与场景阶段对比。

**可得数据前提/口径限制**：
- 需要线程级 on-CPU、wakeup-to-run、runnable delay 的可比口径；不同采样窗口需统一。
- 若仅能获得进程级指标，则该方法退化为“进程关键性”判断，精度下降。
- 调度延迟口径需明确是否包含抢占、睡眠等待等成分，避免跨口径对比。

**方法 B：依赖图中心性判定**
- 基于 Binder/wakeup 构图，中心度高的线程定义为关键线程。
 - 该方法可区分“被依赖”与“主动发起”线程，有助于识别关键路径上的桥接线程。
 - 构图建议以“调用频次 + 时序邻近”作为边权，避免将弱相关调用纳入关键路径。
 - 中心度可结合入度与介数，识别“高依赖汇聚线程”与“链路瓶颈线程”。
 - 对短时突发的高中心性线程需加注释，避免误将“瞬时桥接”视作长期关键。

**可得数据前提/口径限制**：
- 需要可观测的线程间依赖信息（如 Binder 调用链或唤醒关系），且需时序对齐。
- 若缺少调用方向或时序信息，中心性结论可能退化为“共现相关性”。
- 依赖图口径需注明“系统服务是否可见”，否则会低估系统线程的重要性。

**方法 C：突发窗活跃度判定**
- 在 `event_cnt` 突发窗中显著活跃的线程视为关键线程。
 - 适用于短时爆发场景，可揭示关键线程在高压窗口的贡献度。
 - 可将活跃度分为“启动突发”“交互突发”“恢复突发”三类，避免混淆不同阶段。
 - 建议结合帧时间或交互强度，筛掉“非体验相关突发”。
 - 与方法 A/B 交叉验证，可识别“短时关键线程”与“长期关键线程”。

**可得数据前提/口径限制**：
- 需要事件密度与线程活跃度的同窗统计，否则突发窗与线程活跃无法对齐。
- 突发窗定义依赖阈值或分位数，跨场景需重新标定基线分布。
- 若缺少体验侧指标（帧时间、交互强度），难以区分“体验相关突发”与“系统噪声突发”。

**调研补充与实践路径**：
- 关键线程判定可采用“多指标融合打分”，例如 on-CPU、wakeup-to-run、迁核次数、Binder 依赖强度的加权组合。
- 关键线程稳定性可通过“不同场景下的 Top-K 一致性”检验。
 - 对“临时高活跃”线程进行分层标注，避免与长期关键线程混淆。
 - 建议保留“线程角色解释”字段，标注 UI/Render/系统服务/IO 等类别，提升结论可解释性。
 - 对关键线程给出“影响路径描述”，说明其与体验指标（帧时间、卡顿）关联的证据链。

**可行性与权限评估**：
- 可行性：中-高。基于现有调度指标与依赖信息可构建判定框架。
- 权限：中-高。依赖图需要 Binder 或调度链信息，可能要求系统级采集能力。
- 贴合主题：高。关键线程识别是 TracePilot 的核心研究目标之一。

### 3.2 重载/非重载识别方法

**方法 A：PSI 压力指标**
- 通过 `/proc/pressure/cpu|memory|io` 获取资源压力。
  - https://docs.kernel.org/accounting/psi.html
 - PSI 的 some/full 指标可用于区分“部分等待”与“全局阻塞”状态。
 - CPU PSI 偏高常与调度拥塞相关，Memory/IO PSI 偏高更易产生长尾等待。
 - 建议分别统计压力峰值与持续时间，避免仅以瞬时峰值定义重载。

**可得数据前提/口径限制**：
- 需要可访问的 PSI 指标（some/full）与时间窗对齐的采样数据。
- PSI 为系统级压力指标，无法直接定位到线程或进程，需要与调度数据联合解释。
- 不同内核版本对 PSI 的采样粒度与统计口径可能存在差异，应注明版本与采样周期。

**方法 B：窗口级事件密度阈值**
- `event_cnt` 超过 P95 或 P99 定义为重载窗。
 - 建议结合不同场景的基线分布，避免使用统一阈值导致误判。
 - 可引入“事件密度爬升速率”，识别突然恶化与渐进恶化两类重载。
 - 对短时高密度但体验无明显下降的窗口，应标注为“高负载非重载”。

**可得数据前提/口径限制**：
- 需要稳定的事件统计口径（事件类型、去重规则、时间窗长度）。
- 若事件统计来源变更（如新增事件类型），历史阈值失效，应重新估计分位数。
- 事件密度对不同场景敏感，必须以场景内分布为基线。

**方法 C：帧时间与调度延迟联合**
- frame time P95 上升 + runnable delay 上升，视为重载状态。
 - 可补充“jank rate 变化”作为体验侧验证指标。
 - 若帧时间上升而调度延迟不显著，可能为渲染/合成链路瓶颈，不宜归为调度重载。
 - 建议区分“体验重载”（帧时间恶化）与“系统重载”（延迟恶化）两类标签。

**可得数据前提/口径限制**：
- 需要同窗的帧时间统计与调度延迟统计，且二者必须时间对齐。
- 若帧时间来源为应用层统计，需注明是否包含合成/合并阶段，避免口径偏差。
- 调度延迟统计需一致的线程选择范围，否则体验链路与调度链路不匹配。

**调研补充与实践路径**：
- 对重载状态进行“层次化划分”，区分 CPU 压力主导与内存/IO 压力主导。
- 结合调度延迟、PSI 与帧时间的同步性判断“重载是否直接影响前台体验”。
 - 在跨场景对比中区分“慢性重载”与“短时尖峰重载”，增强解释力。
 - 重载判定建议保留“证据链完整度”标注，避免单指标驱动结论。
 - 对重载窗口的关键线程稳定性进行回溯分析，形成“重载-关键线程-体验”的关联说明。

**可行性与权限评估**：
- 可行性：中。方法可实施，但依赖 PSI 与调度事件等系统级指标的可用性。
- 权限：中-高。PSI 与部分调度统计可能仅在系统侧可见。
- 贴合主题：高。重载识别是解释调度异常与体验波动的关键支撑。

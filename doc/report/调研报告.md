# 基于 Android 16 实体机与 eBPF 的预测调度方案调研报告（第二版）

**调研日期：** 2026-04-04  
**适用条件：** 在 **Google 官方支持的 Android 16（或对应 AOSP 版本）实体机**上开展 **预测调度** 与闭环验证。

**修订说明（相对前一版）：**  
- **环境：** 由「虚拟化环境」改为 **实体机（推荐 Pixel + Android 16）**。  
- **采集：** 以 **eBPF** 为主，从内核底层提取 **进程/线程级细粒度事件**，在复杂进程集合中定位 **典型应用** 的 **关键线程与时间序列**。  
- **决策：** 将时间序列交给 **大语言模型（LLM）或自研序列/行为预测模型**，输出 **调度策略/意图**。  
- **执行：** 将策略编码为 **调度指令**，通过 **eBPF 机制**（如 **BPF map**、与 **sched_ext** 协同等）在 **不修改内核源码、不重编内核镜像** 的前提下写入内核可执行路径；与「仅用户态 syscall」方案并列说明。  
- **前一版中的「Perfetto 唯一训练源 + GBDT」路径**不再作为本文主线；若需对照可保留为 **辅助观测（Perfetto）** 或 **消融实验**。

**权限与工程声明：** 在量产 Android 上 **加载自定义 eBPF 程序**通常需要 **root、可解锁引导程序的开发机、或 Google 提供的内核调试能力** 等条件；下文所述「闭环」以 **具备 BPF 加载权限的测试环境** 为前提，论文中须**如实写明设备与权限**。

## 一、核心问题与约束

### 1.1 研究目标

在 **Android 16** 上构建 **预测调度（predictive scheduling）** 闭环，包含以下环节：

1. **底层细粒度采集（eBPF）**  
   使用 **tracepoint / kprobe / BPF map（ringbuf/hash）** 等，采集调度切换、唤醒、Binder IPC、锁竞争等事件，还原 **进程—线程—时间** 维度的可观测序列；在系统进程众多、关系复杂的前提下，通过 **包名 / UID / 前台应用状态** 等过滤，聚焦 **典型应用**（如用户指定 benchmark App 或前台 Top App）。

2. **行为序列构造**  
   将同一 tid / 进程链路上的事件按时间排序，形成 **多变量时间序列**（如：片上时间、等待时间、跨核迁移次数、Binder 延迟分位数等），必要时与 **帧/VSYNC** 等用户态或 ftrace 辅助数据对齐。

3. **预测模型**  
   - **方案 A：** **大语言模型**（或 VLM，若引入多模态）对序列做 **下一窗口行为/瓶颈模式** 预测，输出结构化 **调度建议**（需通过 **prompt + JSON schema** 或微调保证输出可解析）。  
   - **方案 B：** **自研模型**（Transformer / TCN / LSTM 等）直接对张量化序列做分类或回归，输出 **离散策略 ID** 或 **连续参数**（亲和性、uclamp、优先级档位等）。

4. **向内核传递调度指令（不改内核源码）**  
   - 用户态将预测结果编码后写入 **BPF map**；由已加载的 **BPF 程序**（如与 **sched_ext** 结合的调度策略程序，或辅助 **cgroup/affinity** 的协调程序）**读取 map** 并在内核态或受控路径应用。  
   - **「不改写内核」**指：**不修改 AOSP/Linux 内核 C 源码、不重编整树内核**；允许 **加载/更新 BPF 字节码与 map**（仍受内核 BPF 子系统与 SELinux 约束）。

**目标体验指标（可选组合）：** 帧稳定性、jank、冷启动时延、调度尾部延迟等。

### 1.2 实验约束

| 约束 | 说明 |
|------|------|
| **硬件/系统** | **Android 16** 实体机；建议 **Google Pixel** 系列以便与 **GKI / 官方内核文档** 对齐。 |
| **eBPF 权限** | 自定义 BPF 程序加载需 **开发机权限**；无权限时只能采用 **系统已有 eBPF 能力** 或 **降级为 Perfetto + 纯用户态调度**。 |
| **实时性** | LLM 推理延迟可能较大，需 **滑动窗口 + 异步更新** 或 **小模型端侧推理**；自研模型需给出 **推理时延上界**。 |
| **安全** | 调度指令必须经过 **白名单动作空间** 与 **回滚策略**（如 map 清空即恢复默认）。 |

### 1.3 需要验证的可行性

| 维度 | 关键问题 |
|------|----------|
| eBPF 全功能 | Android 16 GKI 是否暴露所需 **sched/binder/futex** 等 tracepoint？**BTF/CO-RE** 是否完备？ |
| 策略入口 | 目标内核是否支持 **sched_ext**？若否，是否仅能通过 **用户态** `sched_setaffinity` / **cgroup** 与 **BPF 辅助**组合？ |
| LLM 可解析输出 | 如何保证输出 **可机器解析** 且 **不越权**？ |
| 论文可信度 | 实体机实验较虚拟环境更易被接受；需报告 **机型、内核版本、权限级别**。 |

### 1.4 数据流与控制流（总览）

```
 eBPF 采集（内核事件 → ringbuf/聚合 map）
        ↓
 典型应用过滤 + 线程级时间序列构造
        ↓
 LLM / 自研序列模型（预测下一窗口调度意图）
        ↓
 调度指令编码 → 用户态写入 BPF map / 触发辅助 BPF 程序
        ↓
 内核侧执行路径（sched_ext / 与 cgroup 协同 / 仅用户态回退）
```

**可选辅助：** **Perfetto** 并行录制用于 **离线对齐、论文图表、与纯 eBPF 对照**，但不作为唯一数据源。

## 二、业界调研（摘要）

- **CPU 调度：** CFS / EAS、**EEVDF** 等仍为主流；**开源**（AOSP / kernel.org）。  
- **eBPF on Android：** 官方文档定义 **扩展内核能力** 的路径；量产机常见 **网络统计** 等内置 BPF。**自定义调度类** 属研究/工程前沿。  
- **sched_ext：** Linux 主线 **BPF 可插拔调度**；Android 是否启用取决于 **内核版本与编译配置**，需在目标机上 **实测**。  
- **学习式调度：** 集群场景有 **Decima** 等；移动端 **LLM + 内核** 尚无标准产品形态，本文属 **探索性架构**。

**立项表述：** 在 **实体机 + 可验证 BPF 权限** 前提下，探索 **「细粒度观测 → 序列预测 → BPF 传递策略」** 相对 **默认调度** 的收益与开销。

## 四、eBPF：底层采集与「无内核源码修改」的策略注入

### 4.1 设计原则

- **采集侧：** `sched_*`、`binder_*`、可选 `futex` 等；**ringbuf** 上报用户态做序列化；必要时 **hash map** 维护 per-tid 统计。  
- **注入侧：** **不修改内核 C 代码**；通过 **已加载 BPF 程序** 读取 **用户态更新的 map**（如 `tid → preferred_cpu / boost / uclamp 提示`）。若内核支持 **sched_ext**，可将 **struct sched_ext_ops** 实现为 BPF 程序，从 map 读策略；若不支持，则 **BPF 仅做观测与辅助**，**实际 CPU 绑定**仍通过 **用户态** `sched_setaffinity` / **cgroup** 完成（需在文中区分「强内核注入」与「用户态回退」）。

### 4.2 总体架构示意

```
┌─────────────────────────────────────────────────────────────┐
│ Android 16 实体机（GKI 内核 + BPF 子系统）                     │
│  ┌───────────────────────────────────────────────────────┐  │
│  │ eBPF 程序（采集）：tracepoint/kprobe → ringbuf / maps │  │
│  └───────────────────────────┬───────────────────────────┘  │
│  ┌───────────────────────────▼───────────────────────────┐  │
│  │ 用户态：典型 App 过滤、时间序列构造、LLM/自研模型推理    │  │
│  └───────────────────────────┬───────────────────────────┘  │
│  ┌───────────────────────────▼───────────────────────────┐  │
│  │ 用户态：将调度指令写入 BPF map（或调用回退接口）         │  │
│  ┌───────────────────────────▼───────────────────────────┐  │
│  │ eBPF 程序（策略）：sched_ext 或辅助路径读 map 生效      │  │
│  └───────────────────────────────────────────────────────┘  │
└─────────────────────────────────────────────────────────────┘
```

### 4.3 采集脚本示例（CO-RE 示意，与内核版本对齐）

```c
// sched_monitor.bpf.c — 调度事件采集（示意）
#include "vmlinux.h"
#include <bpf/bpf_helpers.h>
#include <bpf/bpf_tracing.h>
#include <bpf/bpf_core_read.h>

struct sched_event {
    u32 prev_tid;
    u32 next_tid;
    u32 prev_cpu;
    u32 next_cpu;
    u64 timestamp;
};

struct {
    __uint(type, BPF_MAP_TYPE_RINGBUF);
    __uint(max_entries, 256 * 1024);
} events SEC(".maps");

SEC("tracepoint/sched/sched_switch")
int trace_sched_switch(struct trace_event_raw_sched_switch *ctx)
{
    struct sched_event *e;
    e = bpf_ringbuf_reserve(&events, sizeof(*e), 0);
    if (!e) return 0;
    e->prev_tid = ctx->prev_pid;
    e->next_tid = ctx->next_pid;
    e->timestamp = bpf_ktime_get_ns();
    bpf_ringbuf_submit(e, 0);
    return 0;
}

char LICENSE[] SEC("license") = "GPL";
```

### 4.4 策略 map 示意（用户态写、内核侧 BPF 读）

```c
// policy 由用户态根据 LLM/模型输出写入；调度类 BPF 在 enqueue/select_cpu 等路径查表
struct sched_hint {
    u32 preferred_cpu;
    u32 uclamp_min_hint;
    u32 flags;
};

struct {
    __uint(type, BPF_MAP_TYPE_HASH);
    __uint(max_entries, 4096);
    __type(key, u32);   // tid
    __type(value, struct sched_hint);
} tid_policy SEC(".maps");
```

## 五、预测模型层：LLM 与自研模型

本章说明：在 **eBPF 已得到线程/时间对齐的事件流** 之后，如何构造 **模型输入**、定义 **预测任务**、分别采用 **大语言模型** 与 **自研序列模型** 输出 **可执行的调度指令**，并满足 **实时性、安全性与可复现性**。

### 5.1 预测任务的形式化

将调度辅助建模为 **在滑动时间窗上的决策问题**。设窗口长度 \(W\)（如 100 ms～1 s），步长 \(S\)，在时刻 \(t\) 已观测到某 **典型应用** 内一组线程 \(\mathcal{T}_t\) 的特征序列。

| 任务类型 | 含义 | 输出形态 |
|----------|------|----------|
| **T1：关键线程识别** | 下一窗口内最可能影响流畅度的少数 tid | `topk_tid` 或多标签 `0/1` |
| **T2：策略档位分类** | 在有限集合中选一种「调度模板」（亲和性模板、uclamp 档、优先级档） | 离散类 `k ∈ {1,…,K}` |
| **T3：参数回归** | 对 `preferred_cpu`、`uclamp_min` 等做有界回归 | 连续值 + 裁剪 |
| **T4（可选）：异常/尖峰预测** | 是否将出现 runqueue 长尾、Binder 阻塞尖峰 | `0/1` 或分位数预测 |

**与下游衔接：** T1～T3 的任意组合均可映射为 **§4.4** 中 `sched_hint` 或用户态 `sched_setaffinity` 的输入；论文中应 **固定一种主任务**（如 T2），其余作 **消融**。

### 5.2 从 eBPF 事件到模型输入（特征与张量）

1. **对齐与分窗**  
   对 ringbuf 中事件按 **时间戳排序**，按 **tid** 聚合为多条子序列；对每个窗口截取 \([t-W, t]\) 内数据。缺失线程可填 **0** 或上一窗口值（需在文中声明）。

2. **特征通道（示例，可按实现裁剪）**  
   对每个 `(窗口, tid)` 可构造 **标量特征**：片上时间占比、runqueue 等待分位数、被唤醒次数、跨 CPU 迁移次数、Binder 出站/入站延迟统计等（均由 eBPF 聚合或用户态二次统计）。  
   **注意：** 特征定义须与 **训练、线上一致**，避免「线下用 Perfetto 多列、线上只有 eBPF」的分布偏移（若存在，需 **归一化** 或 **特征对齐层**）。

3. **张量形状**  
   - **自研网络：** 常见输入 `X ∈ ℝ^{B×T×F}`（batch × 时间步 × 特征维）。  
   - **LLM：** 可将每个窗口的统计量与 **tid→comm** 等 **表格化文本** 拼成 **单条 prompt**（见 §5.4），或先经 **小型编码器** 压成短向量再喂给 API（降低 token）。

4. **标签（监督信号，若需训练）**  
   可用 **下一窗口** 的代理指标构造弱标签，例如：若某 tid 与 **高 jank / 高延迟** 共现则标为正样本；或使用 **启发式规则**（高 P95 等待 → 关键）。**自监督**（预测下一时刻特征、对比学习）可作为 **预训练**，再在少量标注上 **微调**。

### 5.3 自研序列模型路径（推荐作「主可复现基线」）

| 结构 | 适用点 | 备注 |
|------|--------|------|
| **TCN / 1D-CNN** | 局部模式、低延迟推理 | 参数量可控，易部署 |
| **LSTM / GRU** | 中长依赖、实现简单 | 注意梯度与实时性 |
| **Transformer Encoder** | 多线程交互、全局依赖 | 计算量较大，可做 **小模型** 或 **蒸馏** |

**训练流程建议：**  
（1）按窗口划分数据集，**按场景/会话** 划分 train/val/test，避免 **时间泄漏**；  
（2）分类任务用 **交叉熵 + 类别权重**（关键线程往往稀疏）；  
（3）回归任务对 `uclamp`、`cpu id` 等做 **sigmoid/softmax 参数化**，避免越界；  
（4）报告 **推理延迟（ms/窗口）** 与 **CPU 占用**，与 **§7** 端到端开销一致。

**与 LLM 关系：** 自研模型可作为 **强基线** 与 **蒸馏教师**（用大模型离线生成伪标签再训练小网络）。

### 5.4 大语言模型（LLM）路径

**适用场景：** 需要 **自然语言解释**、**少样本规则迁移**、或快速 **原型验证**；**不适合**作为唯一实时路径除非 **端侧小模型** 或 **异步更新**。

| 环节 | 做法 |
|------|------|
| **输入构造** | 将窗口内 **各 tid 的统计表** + **应用包名** + **允许的动作说明** 写入 system prompt；可附 **简短内核/调度术语表** 减少胡编。 |
| **输出约束** | 强制 **JSON Schema** 或 **function calling**（若 API 支持），字段与 `tid_policy` **一一对应**；禁止自由文本直写 map。 |
| **校验** | 用户态 **schema 校验** → **tid ∈ 当前前台白名单** → **数值范围裁剪** → 再写 BPF map。 |
| **部署** | **云端 API**：注意 **隐私**（脱敏包名、不上传原始路径）、**时延**；**端侧**：Gemini Nano / 量化 LLM 等需单独评测 **TPS 与耗电**。 |
| **成本** | 在线每窗口调用可能昂贵，可采用 **降频调用**（仅场景切换或 jank 上升时触发）或 **小模型触发 + 大模型兜底**。 |

**风险：** 幻觉导致 **非法 tid 或超大 uclamp** → 必须 **硬校验**；审稿会关注 **可重复性**，建议固定 **prompt 版本号** 与 **温度参数**。

### 5.5 两条路径的对比与组合

| 维度 | 自研序列模型 | LLM |
|------|----------------|-----|
| 延迟 | 易优化至毫秒～十毫秒级 | 通常更高（除非端侧小模型） |
| 可解释性 | 需 SHAP/注意力等 | 天然语言解释（需防编造） |
| 数据效率 | 需一定标注或自监督设计 | 少样本提示强，但输出需强约束 |
| 工程依赖 | PyTorch / ONNX | API 密钥与网络 |

**组合策略（可选）：** **自研模型** 做 **每窗口实时档位**；**LLM** 在 **每日离线** 或 **场景变更时** 生成 **规则模板/特征权重**，写入配置文件，由用户态守护进程加载。

### 5.6 安全动作空间与回滚（模型层必须约定）

- **动作白名单：** 仅允许预先枚举的 **(模板 ID)** 或 **(cpu_mask, uclamp_min) 离散格点**，模型输出先映射到 **最近合法点**。  
- **作用域：** 仅对 **指定 App 的 tid 列表** 生效，避免误伤 `system_server`。  
- **回滚：** 超时未更新 map、或校验失败 → **清空策略 map** 或写入 **默认 neutral 值**，回退 **B1**。

### 5.7 模型层评测指标（与第七章衔接）

除端到端 jank 外，建议在 **验证集**上单独报告：

- **分类：** Precision/Recall/F1（关键线程）、Top-K 命中率；  
- **档位分类：** 准确率、混淆矩阵；  
- **延迟：** p50/p95 **推理时间**、是否超过 **窗口步长 S**；  
- **稳定性：** 连续窗口 **策略翻转次数**（避免抖动）。

```python
def series_to_policy(X, model, mode="torch"):
    if mode == "torch":
        raw = model(X)  # logits 或分布参数
        policy_id = int(raw.argmax(dim=-1))
    elif mode == "llm":
        prompt = tabulate_to_prompt(X)
        raw_json = llm_api.generate(prompt, json_schema=POLICY_SCHEMA)
        policy_id = parse_policy(raw_json)
    policy = ACTION_WHITELIST[policy_id]
    policy = validate_tid_scope(policy, foreground_tids=ALLOWED_TIDS)
    policy = clip_to_bpf_limits(policy, SCHED_HINT_LIMITS)
    bpf_map_batch_update("tid_policy", policy)
    return policy
```

## 六、策略生效路径对比

| 方案 | 机制 | Android 16 上可行性 |
|------|------|----------------------|
| **A：BPF map + sched_ext** | BPF 调度类读 map | **视内核是否编译 sched_ext** |
| **B：BPF map + 用户态应用** | 模型输出 → 用户态 `sched_setaffinity` / cgroup | **一般可行**（权限足够时） |
| **C：仅观测** | eBPF 只采集不注入 | **用于安全基线** |

**建议：** 主文以 **A 或 B** 之一为主路径，另一为 **回退（fallback）**。

# TracePilot 补充调研报告
作者：李松茂
**修订日期**：2026-05-10  
**项目**：TracePilot（基于数据驱动的 Android 任务调度观测与分析）  
**仓库**：[OSH-2026/TracePilot](https://github.com/OSH-2026/TracePilot)  
**设备与主线**：Google Pixel 6a、Android 16、真机 root 后 eBPF 采集；主数据流为 eBPF → ringbuf → 落盘（JSONL/CSV 等）→ 离线分析与报告（与 README 表述一致）。  
**报告性质**：在既有《调研报告》《移动端 eBPF 场景与指标》及仓库已有场景数据基础上，根据**导师补充调研要求**与《TracePilot_调研扩展》初步结论，对「系统行为（网络为重点、传感器为补充）」「高频场景（支付、游戏等）」「关键/非关键与重载/轻载区分」进行系统化补充；并对接仓库 ebpf/ebpf_data 中**已落地的真机数据类型**，给出可执行的扩展路径与口径说明。

## 摘要

本报告完成三件事：（1）将导师提出的**网络优先、传感器可选、高频场景（支付/游戏等）、重载/轻载与关键/非关键区分**等要求，映射到 TracePilot 现有观测栈与数据表结构；（2）基于 GitHub 仓库 ebpf/ebpf_data 目录，**逐项说明当前真机已采集的数据文件类型、语义与可扩展字段**；（3）将《TracePilot_调研扩展.pdf》中的**研究基础、系统行为扩展、高频场景、关键/重载识别方法及预期输出**等要点，融入正文各节并给出**与 PDF 章节的显式对照**（见 §1.3），不另附全文转录。

## 二、仓库真机数据资产总览（ebpf/ebpf_data）

以下依据 GitHub API 与原始文件抽样（2026 年 2 月前后 main 分支结构），对 ebpf/ebpf_data 下各子目录的**数据类型、格式、用途及与扩展调研的衔接点**作说明。目录入口：[ebpf/ebpf_data](https://github.com/OSH-2026/TracePilot/tree/main/ebpf/ebpf_data)。

### 2.1 QQ页面切换场景/

| 文件 | 类型 | 说明与字段语义（摘要） |
|------|------|------------------------|
| behavior_features.csv | CSV | 按**秒级窗口**与 **pkg** 聚合的行为特征表；典型字段含 window_sec、pkg、event_cnt、uniq_cpu、uniq_comm、uniq_tgid（与既有 behavior 分析报告一致）。用于跨包并发与主包突发分析。 |
| behavior_analysis_qq.csv | CSV | 主包（QQ）逐窗明细及突发标注（如 burst 等，见仓库内分析脚本产物）。 |
| behavior_analysis_top_packages.csv | CSV | 按包聚合的 Top 活跃包统计，支撑「系统 UI / Launcher 与主包同量级」类结论。 |
| behavior_analysis_summary.txt | 文本 | 自动化汇总摘要，便于答辩与报告引用。 |
| page_turning_events.log | 日志 | 页面切换场景下经处理的 eBPF 事件流（体积较大），用于追溯与复现。 |
| page_turning_raw.log | 日志 | 原始采集日志，用于对齐与调试。 |

**与导师要求的衔接**：该目录是「**关键场景（页面切换）**」的代表数据；扩展时应在**同一窗口粒度**上增加 net_*、sensor_* 等列，或维护**平行宽表**再按 window_sec 外键关联，避免破坏既有复现脚本。

### 2.2 feed_scroll/（Chrome 信息流滚动）

| 文件 | 类型 | 说明与字段语义（摘要） |
|------|------|------------------------|
| chrome_scroll_topdown.jsonl.gz | gzip 压缩 JSONL | **行级原始/半原始事件**的大体量落盘，适合全量重放与自定义特征重算。 |
| feed_scroll_events_by_second.csv | CSV | **按秒聚合**的调度相关计数；表头示例：window_sec,window_start_ns,window_end_ns,total_events,sched_switch,sched_waking,sched_wakeup,cpu_frequency。可直接与网络按秒聚合表对齐。 |
| feed_scroll_threads_summary.csv | CSV | **线程级汇总**（PDF 中「方法 A：关键线程识别」明确引用该文件口径），含 on-CPU、迁核、wakeup-to-run、runnable delay 等派生量，用于关键线程与重载窗口的联合判定。 |
| chrome_scroll_topdown_framestats.txt | 文本 | **帧相关统计**辅助，用于 jank / frame time 与调度窗对齐（与 PDF 中 Perfetto/JankStats 引用方向一致）。 |
| chrome_scroll_topdown_summary.json | JSON | 会话级或片段级汇总元数据，便于自动化报告入口。 |

**与导师要求的衔接**：已具备「**连续交互**」关键场景与**线程级**指标，是叠加**网络突发标签**与**传感器运动强度标签**的最佳基准之一。

### 2.3 camera/

| 文件 | 类型 | 说明 |
|------|------|------|
| camera.txt | 文本 | 相机场景下采集日志（体积大），属**新场景**扩展的先行数据。 |
| camera_behavior_features.csv | CSV | 与 QQ 场景同构的**按窗 + 包**行为特征表，便于跨场景对比「事件密度、线程多样性、CPU 参与」等。 |

**与导师要求的衔接**：相机场景同时涉及**传感器（物理持机、姿态变化）**与**重负载（编解码/预览线程）**潜力，适合作为「重载/轻载」与「关键线程」方法的**压力对照场景**。

### 2.4 数据流与扩展字段建议（统一宽表思路）

建议在现有 window_sec（或 window_start_ns/window_end_ns）主键上扩展三类系统行为列（与《TracePilot_调研扩展》第 1 章一致）。与 PDF **§1.3 预期输出**一致：扩展后的窗口级表在**事件侧**应继续覆盖 **调度 + Binder + 频率**（与现有 eBPF 管线一致），在**系统侧**叠加 **网络流量 + 传感器状态**，以支撑因果链解释与跨场景可比。

1. **网络（优先）**：`net_bytes_total`、`net_bytes_fg`、`net_bytes_bg`、`net_packets_total`（均为**派生统计口径名**，实现可来自 Android eBPF 流量监控或系统 UID 级计数，见官方文档 [eBPF traffic monitor](https://source.android.com/docs/core/data/ebpf-traffic-monitor?hl=zh-cn)）。除与调度突发窗外，建议按 PDF 形成 **「网络扰动强度」分层标签**，便于跨场景对比。  
2. **传感器（可选，非必须 eBPF）**：sensor_motion_level、sensor_rotation_change、sensor_sampling_rate（**分析型标签**；采集可走 Android `SensorManager` 用户态采样，与内核窗对齐）。若仅实现部分传感器，须在元数据中注明**传感器覆盖范围**（PDF 要求）。  
3. **重载/关键语义标签（派生）**：load_tier（轻/中/重）、thread_criticality_score、scene_critical（0/1）等，由本文第五章所述规则计算写入。

## 三、系统行为扩展：网络为重点，传感器为补充

### 3.1 网络行为（导师强调重点）

**目标**：区分「调度瓶颈」与「网络突发」对卡顿窗的共同或独立贡献，形成「网络扰动 → 调度异常 → 体验波动」证据链（与 PDF 1.1 节一致）。

**实现路径摘要**：

- 以 **UID + 时间窗** 聚合与现有 eBPF 调度窗对齐；区分**前台/后台**流量，避免将系统服务流量误归因为前台应用（PDF 已强调）。  
- 同时观察**字节速率**与**包量变化**，缓解「总字节不大但小包风暴」的误判。  
- 将**网络突发窗**与 event_cnt 或 total_events 突发窗做**时间叠加与先后次序**分析，识别「先网络后调度」或「并行恶化」模式。

**权限与可行性**：与 PDF 一致——若仅使用系统已有 UID 级统计，可行性中高、权限中等偏高；包级细粒度通常要求更高权限。

### 3.2 传感器行为（手持真实交互，可不使用 eBPF）

**目标**：构造「运动强度 / 方向变化」标签，提升「用户真实在交互」相对「后台噪声窗」的可信度（与 PDF 1.2 节一致）。

**实现路径摘要**：

- 典型传感器：加速度计、重力计、旋转向量（见 [Sensors overview](https://developer.android.com/guide/topics/sensors/sensors_overview)）。  
- **用户态采集**即可：固定采样档位，按秒聚合为与 window_sec 对齐的离散标签。  
- 与 PDF 建议一致：并行设置**静止阈值**与**方向变化阈值**；用变化率区分「连续滑动」与「短促点击」类形态。

**与网络联合**：网络突发 + 低运动强度 → 更可能为「后台同步」；高运动强度 + 调度/帧异常 → 更可能为「前台交互链路」问题。

## 四、高频场景扩展：支付、游戏、视频与仓库现状

《TracePilot_调研扩展》已系统覆盖 **支付（微信/支付宝）、视频（抖音/B 站）、游戏（轻量/3D）** 的场景价值、关注指标与可行性（要点已写入 **§1.3 对照表** 与下表）。本报告在**仓库事实**上补充对齐如下：

| 场景类型 | 仓库中已有目录/数据 | 建议新增或强化 |
|----------|---------------------|----------------|
| 连续交互（信息流） | feed_scroll/ | 叠加网络按窗特征；可选传感器标签 |
| 页面切换 | QQ页面切换场景/ | 同上；支付流程中的「页面跳转」可复用同一套聚合脚本 |
| 相机/预览类 | camera/ | 与传感器强相关，适合作为「重载 + 传感器」联合实验 |
| 支付 | （目录级未单列） | 建议新增 ebpf_data/payment/：脚本化「打开付款码/扫码支付 mock」等可重复流程，输出与 behavior_features.csv 同构宽表 |
| 游戏 | （目录级未单列） | 建议新增 ebpf_data/game/：固定一款轻量与一款 3D；关注 PDF 2.3 节 PSI、频率、输入密度与 frame time 联合 |

**关键场景 / 非关键场景（导师要求）** 建议作**显式标签**而非隐含假设：

- **关键场景**：直接影响课题 KPI 的脚本段（如滚动 20 次中的有效滑动段、支付确认前 10 秒等），在元数据 JSON 或单独 scene_segments.csv 中记录 [t_start, t_end, scene_id, critical=1]。  
- **非关键场景**：准备期、切应用、解锁后静置等，critical=0，在分析中**降权**或剔除，避免污染「关键线程」结论。

### 4.1 支付场景（与 PDF §2.1 对齐）

- **价值**：短时高强度、体验阈值敏感；Binder 调用链与系统服务依赖突出，适合验证「关键线程」判定；流程可高度脚本化。  
- **建议关注指标**：首屏/页面加载时间、Binder latency P95、runnable delay P95、wakeup-to-run P95、网络流量突发、关键线程 on-CPU 占比与迁核次数。  
- **分析要点**：比较支付流程阶段，区分 **冷启动链路** 与 **前台渲染链路**；对比 **支付页与确认页** 的负载差异以观察系统服务依赖强度变化（均为 PDF 明确结论）。  
- **可行性与权限**：与 PDF 一致——可重复性中等受应用策略影响；系统级调度与 Binder 观测权限要求中–高。

### 4.2 视频场景（与 PDF §2.2 对齐）

- **价值**：长时间连续负载，利于观察「调度稳定性」与「网络波动」关系；解码/渲染线程负载清晰，便于关键线程与重载判断；具备 **启动期—稳定期—切换期** 分段。  
- **建议关注指标**：frame time P95/P99、jank rate、on-CPU share、cpu migration count、网络流量波动、解码与渲染线程调度延迟差异。  
- **分析要点**：以帧时间与迁核联合变化表征调度不稳定；将 **网络突发与 frame time 时间对齐**，区分 **渲染侧瓶颈** 与 **拉流侧瓶颈**；启动期关注首帧与调度突发的耦合，稳定期关注抖动频率。

### 4.3 游戏场景（与 PDF §2.3 对齐）

- **价值**：高频输入 + 持续渲染，暴露高帧率下调度边界；GPU/CPU 协同与热控、频率调节带来的波动更明显。  
- **建议关注指标**：frame time P95/P99、runnable delay / wakeup-to-run、CPU 频率变化、PSI、高频输入窗口内事件密度变化。  
- **分析要点**：以 **PSI 作系统重载标签** 并与 frame time 联合判断重载下关键线程稳定性；观察 **CPU 频率变化与帧时间尖峰** 的同步性；将 **输入事件密度与 frame time 对齐**，区分输入压力与渲染瓶颈。

## 五、关键 / 非关键线程与重载 / 轻载：与现有数据的映射

本节将《TracePilot_调研扩展》第 3 章方法落实为与仓库文件**可直接对应**的分析口径。

### 5.1 关键线程识别（多法融合）

| 方法 | 核心判据 | 仓库中可直接利用的数据 |
|------|----------|------------------------|
| A：调度强度 + 低延迟 | on-CPU 高且 wakeup-to-run、runnable delay 低 | feed_scroll_threads_summary.csv |
| B：依赖图中心性 | Binder / wakeup 边高频、时序邻近 | *.jsonl / page_turning_events.log 等需解析构建图 |
| C：突发窗活跃度 | event_cnt 突发窗内活跃线程 | behavior_features.csv、behavior_analysis_qq.csv |

**输出**：为每条线程或每个窗维护 criticality_score 与离散标签 key_thread ∈ {0,1}，并记录证据链字段（与 PDF 调研补充一致）。

与 PDF **§3.1 调研补充**一致，建议在实际落地中：（1）采用 **on-CPU、wakeup-to-run、迁核次数、Binder 依赖强度** 等加权融合打分；（2）用 **跨场景 Top-K 一致性** 检验关键线程稳定性；（3）对「临时高活跃」线程单独分层标注，避免与长期关键线程混淆；（4）保留 **线程角色**（UI / Render / 系统服务 / IO 等）与 **影响路径描述** 字段，便于与帧时间、卡顿等指标写清证据链；（5）对「on-CPU 高但延迟亦高」的线程结合场景阶段辨析是否为 **被迫抢占** 而非关键链路。

### 5.2 重载 / 轻载识别（层次化）

| 方法 | 判据摘要 | 数据依赖 |
|------|----------|----------|
| A：PSI | /proc/pressure/cpu|memory|io 的 some/full | 需 root 可读；与窗对齐后作系统级**重载背景**标签 |
| B：事件密度 | event_cnt 超 P95/P99 为重载窗（场景内分布） | behavior_features.csv、feed_scroll_events_by_second.csv 的 total_events |
| C：体验 + 调度联合 | frame time P95 上升且 runnable delay 上升 | chrome_scroll_topdown_framestats.txt + 线程/秒级 CSV |

**区分「重载线程」与「重载窗口」**：窗口级重载可由 B/C 定义；线程级重载建议用 on-CPU 分位数 + 迁移次数 + Binder 权重（见 PDF 3.2 与调研补充）。

与 PDF **§3.2 调研补充**一致，重载判定建议进一步：（1）**层次化**区分 CPU 压力主导与内存/IO 压力主导；（2）用调度延迟、PSI 与 frame time 的**同步性**判断重载是否直接影响前台体验；（3）在跨场景对比中区分 **慢性重载** 与 **短时尖峰重载**；（4）对「事件密度高但体验指标未明显恶化」的窗口标注为 **高负载非重载**；（5）可引入事件密度的 **爬升速率** 区分突变与渐进恶化；（6）为每条重载结论保留 **证据链完整度** 元数据，并对重载窗口做 **重载—关键线程—体验** 回溯叙述。方法 B 的阈值须按**场景内分布**标定，事件类型或去重规则变更时需重新估计分位数。

### 5.3 「轻载线程」与噪声抑制

对「高 CPU 但非体验链路」的线程（如后台编译、日志线程），应通过**场景关键段标签**与**传感器交互置信度**降权，避免误判为关键线程（回应导师「非关键」诉求）。

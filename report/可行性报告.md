# Android 调度优化项目 eBPF 数据采集与处理可行性报告

# 观测层 
作者：邵晨轩，杨子皓
## 1. 报告目的

本文档面向本项目中 eBPF 相关方案的可行性分析，围绕本次会议明确的两个目标展开：

1. 调研 eBPF 在 Android / Linux 内核侧能够采集到哪些类型、哪些层次的数据。
2. 调研如何将 eBPF 采集到的原始数据进行处理、聚合与打包，使其能够作为后续内核调度分析的模型输入。

因此，本报告重点回答以下问题：

1. eBPF 可以从哪些挂载点采集与调度优化相关的底层数据。
2. 这些数据应如何按层次划分，哪些属于核心数据，哪些属于辅助解释数据。
3. 如何将原始事件转化为结构化的节点、边和时间窗样本。
4. 在本项目当前目标下，eBPF 方案的一期重点应聚焦哪些内容。

## 2. 总体结论

综合当前项目目标与工程约束，可以先给出如下结论：

- eBPF 适合承担内核态行为观测、线程依赖关系提取和低开销在线聚合的任务。
- 对“任务调度优化”这一目标而言，最关键的数据不是全量系统事件，而是能够直接描述线程运行、线程唤醒、Binder 依赖和同步阻塞的数据。
- 原始 eBPF 事件通常粒度细、数量大、字段复杂，不适合直接作为模型输入，必须经过标准化、预聚合和结构化处理。
- 一期工作应优先围绕调度事件、Binder 事务、futex 阻塞、CPU 频率与资源压力展开，不宜将网络栈、文件系统、IRQ/softirq 和精确锁 owner 归因写成核心承诺。

## 3. eBPF 可采集数据一览

从本项目需求出发，eBPF 可采集、且与任务调度优化相关的数据主要包括以下几类：

| 数据类别 | 典型数据 / 事件 | 主要意义 |
| :--- | :--- | :--- |
| 基础上下文数据 | `timestamp`、`pid`、`tid`、`comm`、`cpu`、`prio` | 作为所有事件关联、聚合和构图的基础字段 |
| 调度事件数据 | `sched_switch`、`sched_wakeup`、`sched_waking`、`sched_process_fork/exec/exit` | 反映线程何时运行、何时被唤醒、何时迁移和退出 |
| Binder IPC 数据 | `binder_transaction`、`binder_transaction_received`、Transaction ID、同步/异步标记 | 构建跨进程依赖边，分析服务调用链与关键路径 |
| 同步阻塞数据 | `futex` wait / wake、等待时长、等待频率、部分锁竞争现象 | 判断线程未运行是否由阻塞造成 |
| CPU 与调频数据 | `cpu_frequency`、线程所在 CPU、迁核情况 | 分析调度压力与 DVFS 是否匹配 |
| 内存压力数据 | `mm_vmscan_direct_reclaim_begin/end`、部分缺页、内核分配事件 | 区分调度问题与内存压力问题 |
| I/O 阻塞数据 | `block_rq_issue`、`block_rq_complete` | 判断线程变慢是否受块设备 I/O 影响 |
| Android 调度语义相关数据 | 前台 / 后台分组、top-app / foreground / background 组、线程优先级组合关系 | 区分不同线程的调度重要性 |
| 可扩展观测数据 | 网络收发、IRQ/softirq、workqueue、文件系统读写、热节流 | 适合作为扩展分析项，不建议一期作为核心承诺 |

## 4. eBPF 可采集的数据类型与层次

从“内核调度分析需要什么信息”的角度出发，eBPF 可采集的数据建议划分为以下七层。

### 4.1 统一上下文层

这一层是所有事件的公共基础字段，用于后续事件关联、聚合与构图。建议统一采集：

- 时间戳 `ts_ns`
- `pid`、`tid`
- `comm`
- `cpu`
- `prio` / `nice`
- 所属线程组或进程

这一层本身不直接描述性能问题，但它是后续所有调度、Binder、阻塞和资源事件的统一主键。如果缺少这一层，后续很难将不同来源的事件关联成完整链路。

### 4.2 调度层数据

调度层数据是分析线程执行状态与关键路径的基础，主要包括：

- `sched_switch`：记录线程上下文切换事件，包括切换出的线程（prev）、切换入的线程（next）以及切换时的时间戳。负责计算线程的实际 on-CPU 时间、识别抢占情况和 CPU 时间分布，是追踪线程执行状态的最基础事件。

- `sched_wakeup`：记录线程被唤醒事件，包括被唤醒的线程 ID、唤醒时间戳和唤醒线程（caller）。（更稳妥的表述可写为：可记录或关联出唤醒源线程及被唤醒线程，用于建立线程间的唤醒依赖关系。）负责建立线程间的唤醒依赖关系，估计从唤醒到实际获得 CPU 的延迟，是构建线程依赖图的关键事件。

- `sched_waking`：与 `sched_wakeup` 类似，在唤醒发生的更早阶段触发。负责提供更精确的唤醒序列追踪，对于捕捉快速的唤醒链路特别有用。

- `sched_process_fork`：记录进程创建事件，包括父进程 ID 和新建进程 ID 以及创建时间戳。负责追踪进程的生命周期起点和启动阶段的性能特征，用于分析应用启动的调度行为。

- `sched_process_exec`：记录进程执行 exec() 系统调用的事件，包括新执行的程序名称。负责追踪进程的程序加载时刻和阶段性行为变化，用于区分不同阶段的调度压力。

- `sched_process_exit`：记录进程或线程退出事件，包括退出线程的 ID 和退出时间戳。负责完整追踪进程生命周期，用于分析资源释放和后续进程行为。

这类数据可以回答以下问题：

- 哪些线程在频繁占用 CPU
- 哪些线程处于 runnable 但长期未获得调度
- 哪些线程之间存在明显的唤醒依赖
- 线程是否频繁迁核
- 调度延迟是否集中发生在特定线程或特定 CPU 上

对本项目而言，调度层是最核心的数据层之一。

### 4.3 IPC 与依赖关系层

在 Android 中，Binder 是最重要的跨进程通信机制，因此 Binder 相关数据对依赖图构建非常关键。典型可采集内容包括：

- `binder_transaction`：记录 Binder 事务的发起端事件，包括调用方进程/线程、目标服务进程/线程、Transaction ID、是否同步调用、事务的发起时间戳等信息。（即 可记录或结合事件字段与上下文关联出调用方、服务方、Transaction ID 及同步/异步标记等信息。）负责标记跨进程函数调用的起点，追踪调用方对服务方的依赖关系，是构建跨进程依赖图的关键事件。

- `binder_transaction_received`：记录 Binder 事务到达服务端线程的事件，包括接收方线程、Transaction ID 和接收时间戳。负责测量跨进程调用在 Binder 驱动中的传播延迟，配合 `binder_transaction` 可以计算整个 IPC 往返时间。（二者更直接可用于测量请求从发起到服务端接收的传播延迟；计算“往返时间”，则通常还需要额外关联响应返回阶段事件。） 



这类数据可以用于识别：

- 哪些进程或线程是核心服务依赖点
- 哪些 Binder 链路存在高延迟
- 异步事务是否形成后台隐形负载
- 跨进程依赖是否影响关键线程响应

对 Android 调度优化来说，Binder 层是区别于一般 Linux 调度分析的关键层。



### 4.4 同步阻塞层

这类数据用于判断线程“没有运行”究竟是因为未被调度，还是因为自身被阻塞。典型包括：

- `futex` 等待与唤醒事件
- 等待开始时刻、结束时刻和累计等待时长
- 某线程的等待频率、最长等待时间
- 少量可稳定获取的锁竞争现象

这类数据适合用于识别：

- 线程无法运行是否源于同步阻塞
- 关键线程是否频繁被 futex 或锁等待卡住
- 是否存在长期竞争热点

需要注意的是，eBPF 更适合稳定获取“等待现象”和“等待时长”，但对于“锁持有者是谁”的精确归因往往需要额外推断，一期不宜承诺过满。

### 4.5 资源与运行环境层

此类数据主要用于区分“调度问题”和“资源不足问题”，典型包括：

- `cpu_frequency`
- 线程所在 CPU 与迁核信息
- `mm_vmscan_direct_reclaim_begin/end`
- 块设备 I/O 事件，如 `block_rq_issue`、`block_rq_complete`
- 必要时可参考系统级压力指标，如 CPU / Memory / IO stall

这类数据可以用于判断：

- 高优先级线程是否长期运行在较低频率下
- 当前调度压力与 DVFS 状态是否匹配
- 线程变慢是否由内存回收压力导致
- 是否存在明显的 I/O 阻塞背景

这类数据不直接构成依赖边，但对后续归因和解释非常重要。

### 4.6 Android 调度语义层

除了内核事件本身，还需要关注线程所处的 Android 调度语义环境，例如：

- 前台 / 后台进程组
- top-app、foreground、background 等任务组
- 线程优先级与调度组的组合关系

这一层不一定完全通过 eBPF 直接获得，但应在后续处理链路中作为关联字段纳入。原因在于：同样的调度事件，发生在前台关键线程和后台服务线程上，其分析意义并不相同。

### 4.7 扩展层

从技术上讲，eBPF 还可以扩展采集：

- 网络收发与网络栈事件
- 硬中断、软中断、workqueue 事件
- 文件系统读写和缓存相关事件
- 热节流与温度相关事件

这些数据在全量性能分析平台中有价值，但对当前项目的一期目标不是核心，适合作为扩展方向或 Future Work。

## 5. 本项目最关键的数据项

本项目的目标不是构建通用 Android 性能诊断平台，而是为后续的内核调度分析提供可用输入。因此，最关键的数据应满足以下条件：

- 能直接描述线程依赖关系
- 能解释线程为何获得或未获得 CPU
- 能帮助识别关键路径和阻塞路径
- 能在当前实验环境中稳定获取

基于上述标准，建议将重点数据分为三层。

### 5.1 核心数据

以下数据建议作为一期 eBPF 主采集项：

| 数据类型 | 典型事件 | 主要用途 |
| :--- | :--- | :--- |
| 线程切换 | `sched_switch` | 计算 on-CPU 时间、识别抢占和执行分布 |
| 线程唤醒 | `sched_wakeup`、`sched_waking` | 构建唤醒依赖边，估计唤醒到运行延迟 |
| Binder 事务 | `binder_transaction`、`binder_transaction_received` | 构建跨进程依赖边，分析服务调用延迟 |
| 基础上下文 | `pid/tid/comm/cpu/prio/timestamp` | 作为所有事件的统一基础字段 |

这四类数据构成后续构图与关键路径分析的核心基础。

### 5.2 增强数据

以下数据建议作为增强特征或辅助解释项：

| 数据类型 | 典型事件 | 主要用途 |
| :--- | :--- | :--- |
| futex 等待 / 唤醒 | syscall tracepoint | 识别线程被同步阻塞的程度 |
| CPU 频率变化 | `cpu_frequency` | 判断调频背景与调度压力是否匹配 |
| 进程生命周期 | `sched_process_fork/exec/exit` | 分析启动阶段与阶段性行为变化 |
| 内存回收压力 | `mm_vmscan_direct_reclaim_begin/end` | 区分调度问题与内存压力问题 |
| 块 I/O 聚合延迟 | `block_rq_issue/complete` | 判断系统是否存在 I/O 背景瓶颈 |

这些数据主要用于解释“为什么慢”，而不是单独定义主问题。

### 5.3 扩展数据

以下数据适合作为 Future Work，不建议在一期可行性报告中写成必做项：

- 网络 I/O 与网络栈全链路分析
- 文件系统与页缓存效率分析
- 热节流与热源因果归因
- 全量 IRQ、softirq、workqueue 深度分析
- 精细到具体 owner 的锁竞争归因

## 6. 数据处理目标与推荐链路

本项目的数据处理目标已经在会议中明确：将 eBPF 采集到的原始数据，经过处理后形成可用于内核调度分析的模型输入。

由于 eBPF 原始数据颗粒度细、事件量大、字段复杂，不能直接上模型，因此建议采用“采集 -> 标准化 -> 预聚合 -> 结构化输出”的链路。

### 6.1 原始事件采集

建议优先使用 tracepoint 采集结构稳定、语义明确的系统事件；只有在 tracepoint 无法覆盖时，再补充少量 kprobe。

内核态采集到的原始事件，建议尽量统一为如下字段：

| 字段 | 含义 |
| :--- | :--- |
| `ts_ns` | 事件时间戳 |
| `event_type` | 事件类型 |
| `pid` | 进程 ID |
| `tid` | 线程 ID |
| `cpu` | 所在 CPU |
| `comm` | 线程名 |
| `src_tid` | 源线程或调用方线程 |
| `dst_tid` | 目标线程或被依赖线程 |
| `duration_ns` | 持续时间或等待时间 |
| `flags` | 附加标志位 |
| `prio` | 优先级信息 |

统一标准化的目的，是让调度、Binder、futex 和资源压力事件能够进入同一条后处理管线。

### 6.2 内核态预聚合

为降低对系统性能的影响，不建议逐条上报所有 raw event，而应优先在内核态使用 BPF Map 做预聚合，例如：

- 按 `tid` 聚合：
  - `oncpu_ns`
  - `runnable_delay_ns`
  - `wakeup_count`
  - `futex_wait_ns`
- 按 `(src_tid, dst_tid, edge_type)` 聚合：
  - `count`
  - `total_ns`
  - `max_ns` 或 `p95_ns`
- 按 `cpu` 聚合：
  - 频率驻留
  - 调度压力
- 按时间窗缓存最近一段关键事件：
  - 用于异常触发式导出

这一步的目标是减少数据量、降低采集开销，并让用户态拿到的已经是“可分析信号”而不是全量噪声。

### 6.3 用户态结构化处理

用户态接收聚合结果后，应将其转换为适合调度分析和后续策略模块使用的标准格式。建议输出三类结构化结果：

1. 节点表 `nodes`
2. 边表 `edges`
3. 时间窗样本表 `samples`

#### 节点表

用于表示线程或进程的局部状态：

- `window_id`
- `tid`
- `pid`
- `comm`
- `oncpu_ns`
- `runnable_delay_ns`
- `futex_wait_ns`
- `binder_in_count`
- `binder_out_count`
- `cpu_freq_mean`
- `reclaim_time_ratio`
- `io_delay_ratio`

#### 边表

用于表示线程间依赖关系：

- `window_id`
- `src_tid`
- `dst_tid`
- `edge_type`
- `count`
- `total_ns`
- `p95_ns`

其中 `edge_type` 可优先统一为：

- `WAKEUP`
- `BINDER_CALL`
- `FUTEX_WAIT`
- `RUN_ON_CPU`

#### 样本表

用于后续调度分析模型或在线策略模块：

- `window_id`
- `feature_1 ... feature_n`
- `label` 或 `state`

在当前阶段，更建议将其理解为“状态观测向量”或“调度分析输入”，而不是必须带严格监督标签的训练样本。

### 6.4 与内核调度分析模型的关系

结合上述链路，建议从 eBPF 结果中提取以下关键特征：

- `oncpu_ns`
- `runnable_delay_ns`
- `wakeup_count`
- `wakeup_delay_p95`
- `binder_call_count`
- `binder_latency_p95`
- `binder_async_ratio`
- `futex_wait_ns`
- `cpu_freq_mean`
- `cpu_migration_count`
- `reclaim_time_ratio`
- `io_delay_ratio`

这些特征既可以用于：

- 构建线程关键性评分
- 识别关键路径节点
- 判断性能异常的主要来源
- 为后续内核调度分析与策略模块提供轻量状态输入

## 7. 对本项目的一期建议

结合工程复杂度、可移植性和报告可信度，建议本项目一期方案聚焦以下范围：

### 7.1 建议纳入一期的 eBPF 采集内容

- `sched_switch`
- `sched_wakeup` 或 `sched_waking`
- `binder_transaction`
- `binder_transaction_received`
- `futex` 等待 / 唤醒统计
- `cpu_frequency`
- `mm_vmscan_direct_reclaim_begin/end`

### 7.2 建议作为辅助或背景观测的内容

- `sched_process_fork/exec/exit`
- `block_rq_issue/block_rq_complete`

### 7.3 建议暂不作为一期核心承诺的内容

- 全量网络栈观测
- 文件系统缓存效率分析
- 热点与热节流因果归因
- 精确锁持有者识别
- 大规模 kprobe 覆盖方案

## 8. 可行性分析结论

综合当前项目目标与数据处理需求，可以得出以下结论：

### 8.1 技术可行性

本项目中的 eBPF 方案具备明确的技术可行性。其原因在于：

- 调度、Binder、futex、cpu_frequency、内存回收等关键事件均有较成熟的采集路径。
- eBPF 适合承担低开销、可编程、按需聚合的内核态观测角色。
- 通过 tracepoint + 少量必要的 kprobe，可以覆盖本项目所需的大部分关键数据。

### 8.2 数据处理可行性

虽然 eBPF 原始事件复杂且数量大，但通过“统一字段标准化 + 内核态预聚合 + 用户态结构化打包”的处理链路，可以将其转化为可用于内核调度分析的模型输入。也就是说，数据复杂性主要是工程处理问题，而不是方法不可行问题。

### 8.3 工程可行性

本项目若将 eBPF 范围控制在“依赖关系提取 + 调度背景观测 + 轻量聚合”这一层，工程复杂度是可控的。若扩展到网络栈、文件系统、热分析和锁 owner 精确归因，则实现复杂度与验证成本会显著上升，不利于一期落地。

### 8.4 最终结论

对本项目而言，eBPF 方案是可行且必要的。其核心价值不在于采集一切可能的底层数据，而在于围绕调度分析目标，抽取最关键的内核事件，并通过内核态预聚合和用户态结构化处理，将复杂原始事件转化为可用于后续调度分析的模型输入。

## 9. Perfetto 在本项目中的作用

虽然本报告重点讨论 eBPF 数据采集与处理，但从整体数据链路看，仍有必要说明 Perfetto 在项目中的使用方式以及它与 eBPF 的配合关系。

### 9.1 Perfetto 具体怎么采

对本项目而言，Perfetto 更适合作为离线 trace 采集工具，建议采用“配置文件 + 命令行触发 + trace 落盘 + 本机解析”的方式。

在 Android 设备侧，典型流程如下：

1. 编写统一的 Perfetto 配置文件 `.pbtxt`
2. 通过 `adb shell perfetto` 按配置采集指定时长的 trace
3. 将生成的 trace 文件从设备拉回本机
4. 使用 Perfetto 提供的 trace processor 或 SQL 接口做离线解析

建议的执行流程可概括为：

- 在 Pixel 6 上准备统一配置文件
- 用脚本触发具体 benchmark 场景，如冷启动、滑动、切换应用
- 每次场景执行时同步开始和结束 trace
- 采集结束后统一导出 `.pftrace` 或等价 trace 文件
- 在主机端解析出调度、Binder、帧时序和关键线程行为信息

这种方式的优点是：

- 采集流程稳定
- 可以多次复现实验
- 便于不同场景、不同版本策略之间做横向对比
- 后续解析脚本可以固定，不需要每次重新设计采集逻辑

### 9.2 本项目最该开启的 trace 配置

围绕“任务调度优化”这一目标，不建议一开始就开启过多类别，而应优先保证以下几类轨道：

#### 必选配置

- 调度相关事件：
  - `sched_switch`
  - `sched_wakeup`
  - `sched_waking`
- Binder 相关事件
- 线程 / 进程信息
- CPU 频率相关信息
- `atrace` 中与应用关键阶段相关的标记
- `FrameTimeline`

这部分是最核心的，因为它们能够共同回答：

- 线程何时被唤醒、何时真正运行
- 跨进程依赖链路如何传播
- 某个卡顿窗口内前台线程是否被延迟调度
- 问题是否与 Binder 调用或帧渲染链路重合

#### 建议开启的增强配置

- `futex` 或可解析的同步阻塞相关事件
- 内存压力与 reclaim 相关事件
- 必要时的 I/O 相关事件

这部分主要用于解释问题来源，例如区分：

- 是调度器没有及时给 CPU
- 还是线程被锁、内存压力或 I/O 拖慢

#### 不建议一期默认全开的配置

- 全量网络栈
- 全量文件系统
- 全量 IRQ / softirq / workqueue
- 大量细粒度 debug 轨道

原因在于：

- trace 体积会迅速膨胀
- 解析复杂度显著增加
- 会稀释本项目“调度优化”主线

因此，本项目的 Perfetto 配置建议遵循“先少而关键，再按问题增量扩展”的原则。

### 9.3 Perfetto 与 eBPF 如何配合写进报告

在报告中，建议将 Perfetto 与 eBPF 明确写成“分工互补”关系，而不是二选一关系。

推荐表述如下：

- Perfetto 负责离线 trace 采集、跨层时间线对齐和问题回放分析
- eBPF 负责内核态细粒度观测、关键事件增强采集和在线低开销聚合

换句话说：

- Perfetto 更适合回答“问题发生时，系统全局链路是什么样”
- eBPF 更适合回答“内核内部关键线程、关键依赖和阻塞细节到底是什么”

在报告撰写中，建议这样分工描述：

1. 在“数据采集”部分写：
   - Perfetto 负责离线采集调度、Binder、帧时序和应用关键阶段信息
   - eBPF 负责补充更细粒度的内核事件，如唤醒关系、Binder 事务细节、futex 阻塞和资源压力

2. 在“数据处理”部分写：
   - Perfetto 提供跨层可对齐的 trace 主线
   - eBPF 提供更细粒度的内核特征，并在内核态先做预聚合
   - 两者最终都服务于后续调度分析输入的构建

3. 在“方案价值”部分写：
   - Perfetto 保证可以从系统全景角度分析卡顿、慢启动和关键路径
   - eBPF 保证可以从内核细节角度进一步解释线程为什么没有及时运行

这样写的好处是：

- 能避免“Perfetto 和 eBPF 功能重复”的质疑
- 能说明 eBPF 的独特价值在于增强内核态观测，而不是简单重复 trace
- 能让后续系统方案自然形成“Perfetto 看全景、eBPF 看细节”的组合结构

## 10. 建议写入总报告的总结表述

可在总报告中采用如下总结性表述：

> 本项目中，eBPF 主要承担内核态在线观测、线程依赖关系提取和低开销聚合的任务。围绕任务调度优化目标，最关键的 eBPF 数据包括线程切换、线程唤醒、Binder 事务、futex 阻塞、CPU 频率变化以及内存和 I/O 压力等信息，其中调度事件与 Binder 依赖关系是构建关键路径和调度分析模型输入的核心。原始 eBPF 事件不直接作为模型输入，而是通过统一字段标准化、内核态预聚合和用户态结构化打包，转化为节点表、边表和时间窗样本等可用格式，供后续内核调度分析使用。

**本机：** **Google Pixel 6a**，**Android 16**，**Magisk** root。下文 **「本机」** 同此指代。

**课题约束：** **不修改 AOSP/Linux 内核 C 源码、不重编整树内核镜像**；允许在运行时 **加载/更新 eBPF 字节码与 BPF map**（仍受 **内核 BPF 子系统、SELinux、Linux Capabilities、verifier** 约束）。

# 模型层
作者：贺小轩

**文档版本：** 2026-04-14  
**上游输入：** [eBPF可行性报告.md](eBPF可行性报告.md)  
**下游约束：** [执行步骤可行性报告.md](执行步骤可行性报告.md)  
**适用范围：** Android 16 实体机环境下，验证预测模型层是否能够承接 eBPF 结构化输出，并向执行层输出可落地的调度决策。

---

## 1. 阶段定位

本报告位于项目链路中间层，承担“结构化观测数据 → 可预测决策 → 可执行策略”的转换职责。其前一阶段负责从 eBPF 事件中提取结构化样本，后一阶段负责将模型输出映射为 map、affinity、taskset、sched_ext 或离线评估结果。

### 1.1 本阶段输入

- **nodes**：线程或进程在窗口内的状态向量
- **edges**：唤醒、Binder、阻塞等依赖边
- **samples**：窗口级样本或状态样本

### 1.2 本阶段输出

- 关键线程识别结果
- 策略档位分类结果
- CPU 亲和或 uclamp 参数
- 异常或尖峰预测结果

### 1.3 目标对象

模型主要预测下一时间窗的调度倾向，而非用户层动作本身。用户行为仅作为系统负载来源，其直接建模对象是线程级调度状态、关键路径与策略选择。

### 1.4 本阶段目标

本阶段的目标不是构建通用预测模型，而是构建一个能够服务调度优化的、可解释、可约束、可回写的模型层。该模型层需同时满足三项要求：

1. 输入可以稳定构造；
2. 输出可以稳定约束；
3. 输出可以稳定映射到执行层。

---

## 2. 工作规划

### 2.1 总体路线

预测模型层的工作应按以下顺序推进：

1. 固化样本格式，明确单样本表示单 tid 单窗口状态，还是多 tid 联合窗口状态。
2. 建立统一特征清单，将 eBPF 输出映射为固定字段。
3. 设定主任务，优先选择关键线程识别或策略档位分类。
4. 先训练小模型，验证数据是否存在可学习信号。
5. 再加入输出约束，将结果限制在白名单动作空间。
6. 最后接入执行层，先离线回放，再进入 map、affinity 或 syscall。

### 2.2 样本组织方式

每个训练样本宜由三部分组成：

- **上下文**：应用名、场景名、窗口起止时间
- **序列特征**：tid 特征、节点特征、边特征
- **监督目标**：下一窗口标签或策略收益

样本来源应为上一阶段 nodes、edges、samples 的再组织结果，不宜重新回到原始事件流。这样做可以保证输入稳定、维度稳定、含义稳定。

### 2.3 特征清单规划

建议将特征划分为三组：

- **线程状态特征**：`runtime_ms`、`wait_ms`、`migrate_cnt`、`prio`、`cpu_id`
- **依赖关系特征**：`binder_p95`、`binder_count`、`wake_cnt`、`futex_wait_ns`
- **环境背景特征**：`cpu_frequency`、`reclaim_time_ratio`、`io_delay_ratio`

特征设计应优先满足三个条件：

1. 可由 eBPF 或其后处理稳定获得；
2. 训练与线上保持一致；
3. 与调度动作存在可解释关联。

### 2.4 标签设计规划

标签应围绕调度目标设置，而不是围绕原始事件本身设置。可分为三类：

1. **分类标签**：关键线程是否出现、策略档位类别
2. **回归标签**：`preferred_cpu`、`uclamp_min` 等参数
3. **代理标签**：高 jank、高等待、长尾阻塞等风险信号

若真实标注不足，可先采用启发式弱监督，再逐步精化；若后续具备更完整日志，可再引入人工标注或自监督预训练。

---

## 3. 数据到模型的映射

### 3.1 输入形状

常用输入形式为：

- 批次 × 时间步 × 特征
- 批次 × 线程 × 时间步 × 特征
- 批次 × 展平特征向量

### 3.2 特征工程路径

eBPF 侧提供基础统计，用户态完成二次聚合。建议保留三类输入：

- **nodes 特征**：运行时间、阻塞时间、优先级、CPU 亲和
- **edges 特征**：唤醒关系、Binder 关系、阻塞关系
- **samples 特征**：窗口级汇总统计和场景标签

输入应经过压缩、对齐、归一化和缺失填充，避免直接使用原始内核流。这里的关键不在于是否能够张量化，而在于每一维是否有稳定语义。

### 3.3 任务与张量关系

- **关键线程识别**：tid 为主维度，时间为次维度
- **窗口级策略分类**：窗口为主样本，tid 特征做聚合
- **局部行为解释**：保留事件级统计，避免过早平均化

张量形状应服从任务定义，而非预设固定 shape 再反向适配任务。若任务重点不同，输入组织方式也应随之变化。

### 3.4 标签构造方法

标签宜按以下方式构造：

1. **分类任务**：从下一窗口是否出现高 jank、关键线程或策略切换构造标签。
2. **回归任务**：从历史调优结果、规则策略或窗口收益构造 `preferred_cpu`、`uclamp_min` 等目标值。
3. **代理任务**：从性能退化迹象构造风险信号，用于提前预警。

标签构造的核心要求是：窗口对齐明确、定义固定、可重复生成。

### 3.5 训练前检查

训练前至少应检查四项：

- 特征维度是否稳定
- 缺失值是否统一填充
- 标签是否与窗口对齐
- 场景划分是否混杂

若上述条件未满足，模型易出现离线收敛、上线失效的问题。

---

## 4. 模型方案

### 4.1 自研序列模型

**可行性：高。** 该类任务与常见时序预测问题一致，数据结构清晰、目标明确、部署路径稳定，适合输出关键线程分类、策略档位分类、有界回归参数和异常预测。

#### 建议模型

- **端侧实时性优先**：TCN 或小型 LSTM
- **跨线程交互建模**：Transformer Encoder
- **初期验证信号**：MLP 或线性分类器

#### 训练与部署要点

1. 先用简单模型验证数据是否可学。
2. 再增加序列长度或多线程交互建模能力。
3. 对输出层增加范围约束，避免直接产生非法动作。
4. 部署时优先保证推理时延和稳定性，而非追求过大模型容量。

#### 与执行层衔接

输出宜直接对应 `sched_hint`、CPU mask、`uclamp` 或策略 ID，确保模型输出与执行动作一一映射。

### 4.2 LLM / 结构化提示模型

**可行性：中。** 适合作为解释与建议层，不宜作为唯一实时执行源。其可行前提是强约束输出，并将输出限定在固定动作空间内。

#### 使用方式

- 采用 JSON schema 或 function calling
- 动作空间白名单化
- 数值裁剪与 tid 白名单检查
- 降频调用或离线辅助

#### 适用输出

- 规则模板建议
- 场景切换调度建议
- 离线伪标签生成

#### 与执行层衔接

LLM 输出应直接对接执行步骤报告中的固定 schema，不宜额外引入新的字段体系，否则会增加转译成本和出错概率。

### 4.3 两条路径的关系

自研模型与 LLM 并非替代关系，而是分工关系：

- 自研模型负责实时、稳定、可部署的主路径；
- LLM 负责规则生成、策略解释和离线辅助。

若项目目标偏向论文复现与闭环验证，自研模型应作为主线，LLM 作为补充路径。

---

## 5. 实施节奏

### 5.1 第一阶段：样本与特征固化

目标是形成稳定的训练输入，而非直接追求高精度。

工作内容包括：

- 固化窗口长度和步长
- 固化 tid 选择规则
- 固化节点、边、样本三类数据结构
- 固化特征命名、单位和缺失处理方式

输出应为可复用的数据集版本，而不是临时脚本结果。

### 5.2 第二阶段：基线模型验证

目标是确认数据是否包含可学习信号。

优先验证：

- 关键线程是否可分类
- 策略档位是否可区分
- 回归参数是否可收敛
- 连续窗口是否存在稳定趋势

此阶段模型越简单越好，重点是确定“是否值得继续做”，而不是追求极致指标。

### 5.3 第三阶段：输出约束与策略编码

目标是把模型结果转化成执行层可以接受的形式。

应完成以下工作：

- 定义白名单动作空间
- 约定 CPU mask、uclamp、策略 ID 的编码规则
- 定义非法输出回退逻辑
- 定义更新频率和失效回滚条件

这一阶段的重点是可执行性，而不是模型结构本身。

### 5.4 第四阶段：离线评测与系统验证

目标是验证模型输出是否真正改善系统行为。

评测应分为三层：

1. **离线预测评测**：分类准确率、回归误差、Top-K 命中率
2. **窗口级稳定性评测**：策略翻转次数、抖动率
3. **系统收益评测**：jank、帧稳定性、冷启动时延

若离线指标较高而端到端收益较低，通常表明特征与执行动作未对齐、输出动作过细或评测场景存在分布偏差。

---

## 6. 风险与约束

| 风险 | 影响 | 缓释方式 |
|------|------|----------|
| 特征与线上不一致 | 无法部署 | 统一训练/推理特征定义 |
| 标签噪声大 | 学到伪相关 | 采用弱监督和消融实验 |
| 模型延迟过高 | 影响实时闭环 | 使用小模型、异步更新、降频推理 |
| 输出越界或幻觉 | 生成非法策略 | 白名单 + schema 校验 + 回滚 |
| 样本不足 | 泛化能力弱 | 场景分割、数据增强、自监督预训练 |

预测模型层的主要风险不在于“模型不会训练”，而在于“训练结果无法稳定映射回系统动作”。因此，输入定义、标签定义、输出动作和回写路径必须保持稳定。

---

## 7. 结论

预测模型层可行，且处于 eBPF 采集处理与执行落地之间。eBPF 侧已可提供足够细粒度的结构化中间数据，执行步骤报告中也已定义了 map、affinity、taskset、sched_ext 等执行路径，因此本阶段的关键问题集中在输入组织、标签定义、输出约束与系统收益验证。

工程上，本阶段应优先完成三项工作：

1. 样本格式固定
2. 主任务固定
3. 输出动作固定

只要这三项稳定，后续无论是更换 TCN、LSTM、Transformer，还是引入 LLM 作为辅助路径，整体框架均可保持不变。

---

| 字段 | 内容 |
|------|------|
| 依据输入 | eBPF 线程级事件流、nodes / edges / samples、典型应用过滤 |
| 主要模型 | TCN / LSTM / Transformer / LLM 约束输出 |
| 主要输出 | 关键线程、策略档位、CPU 亲和、uclamp、异常预测 |

| 与执行层关系 | 输出映射至执行步骤报告中的 BPF map、affinity、syscall 或离线评估 |


# 执行层
作者：李松茂
## 1. 「执行」层：两种并列路径（核心定义）

本课题所说的 **「执行」**，指在 **观测 → 预测** 之后，把模型输出的 **策略** 落实为 **可执行的调度指令**。该层 **并列** 两种实现，实验与论文中应 **同时说明**，并按本机实测选择 **单一路径、或 S4a+S4b 组合**。**存在** 固定型号真机上 **路径一与路径二均无法完成闭环** 的可能；若发生，按 **§5 保险路径** 处理（**纯观测评估、应用内执行、换载体、或放宽不重编约束**）。

### 1.1 路径一（首选）：调度指令 → eBPF 机制 → 内核可执行路径

**目标：** 将策略编码为 **调度指令**（如 per-tid 亲和性掩码、uclamp/优先级档位、策略 ID 等），通过 **eBPF 机制** 在 **不修改内核源码、不重编内核镜像** 的前提下，进入 **内核可执行路径**（例如调度决策相关逻辑可读 **BPF map**；若内核支持 **sched_ext**，则由 **BPF 调度类程序** 与 map **协同**）。

**典型数据流（逻辑顺序）：**

1. **用户态** 将 S3 输出的策略 **编码** 为定长结构体，写入 **BPF map**（如 `BPF_MAP_TYPE_HASH`，key=tid，value=调度提示）。  
2. **已加载的 BPF 程序** 在内核侧 **attach** 在 **tracepoint/kprobe**（采集）或 **sched_ext 的 ops**（若可用）等路径上，**读 map** 并影响 **是否/如何在可执行路径上采纳该提示**（具体能力取决于 **程序类型** 与 **内核配置**）。  
3. **与 sched_ext 协同（可选）：** 若本机 **`CONFIG_SCHED_CLASS_EXT=y`** 且允许加载 **struct sched_ext_ops** 对应的 BPF 调度器（社区常称 **scx_***），则 **map 中的 tid 级策略** 可在 **enqueue、select_cpu** 等回调中 **被读取**；若 **未启用 sched_ext**，则 **不得** 将课题表述为「已替换 CFS」，应改为 **「map 仅作侧车/观测 + 实际绑核走路径二」**。

**策略 map 示意（libbpf 风格；用户态根据 S3 输出 `bpf_map_update_elem`，内核侧 BPF 在调度相关路径 `bpf_map_lookup_elem`）**

以下为 **定长 value** 的 **HASH map** 定义示例；**字段、对齐、`max_entries`** 以实现与 **verifier** 为准，可随课题裁剪。

```c
// policy 由用户态根据模型输出写入；调度类 BPF 在 enqueue/select_cpu 等路径查表
struct sched_hint {
    u32 preferred_cpu;
    u32 uclamp_min_hint;
    u32 flags;
};

struct {
    __uint(type, BPF_MAP_TYPE_HASH);
    __uint(max_entries, 4096);
    __type(key, u32);   // tid
    __type(value, struct sched_hint);
} tid_policy SEC(".maps");
```

**「不改内核源码」在本路径中的含义：**  
不改变 **内核树中 `.c` 调度器实现**；通过 **bpf(2)** 注入 **已编译的 .o 字节码** 与 **map**。这与 AOSP **开机加载 `/system/etc/bpf/`** 的 **bpfloader** 是 **不同入口**（研究用临时加载更依赖 **root + SELinux 策略**）。

**硬条件（必须本机实测，不能用文档代替）：**

| 条件 | 说明 |
|------|------|
| `CONFIG_BPF_SYSCALL` 等 | 无则无法 `bpf()`。 |
| `CONFIG_DEBUG_INFO_BTF`（常见） | 影响 **CO-RE** 与跨版本可移植性；缺失时倾向 **固定本机内核** 编译 BPF。 |
| `CONFIG_SCHED_CLASS_EXT` | **sched_ext 路径** 的前提；缺失则 **路径一** 只能做到 **「观测 + map 侧车」**，**真正 CPU 亲和性**须由 **路径二** 完成。 |
| SELinux / 能力 | 零售 Android 上 **任意进程** 随意 `prog_load` 常被拒；**Magisk root** 可缓解但仍可能需 **`setenforce 0`（仅实验环境）** 或 **自定义 sepolicy**。 |
| verifier | 调度类 BPF 程序可能 **Verifier 拒绝**；需保留 **dmesg** 与错误码。 |

### 1.2 路径二（并列）：仅用户态 syscall / 工具 — 不改内核、不依赖自定义调度类 BPF

**目标：**  
在不加载 **自定义 sched_ext 调度器**、或 **BPF 加载失败** 时，仍能用 **用户态** 把 **调度指令** 作用到 **线程/进程**，与路径一 **并列** 作为 **可执行闭环**。

**典型手段（示例）：**

- **`sched_setaffinity(2)`** / **`sched_setscheduler(2)`** / **`setpriority(2)`**（对目标进程/线程通常需 **root 或同 uid**）；  
- 命令行 **`taskset`** 等（在 **adb shell su** 或本机 root 进程内调用）；  
- 若 **`cgroup` v2** 挂载且策略允许，可对 **CPU 权重** 做实验（**ROM 差异大**，需实测）。

**与路径一的关系：**

- **并列：** 同一套 S3 输出可先 **写 map**（路径一），再 **由用户态守护进程读回显式调用 syscall**（路径二），或 **仅路径二**。  
- **论文表述：** 路径二 **不是**「BPF 写入内核可执行路径」意义上的 **内核协同调度类**，应明确为 **「用户态调度动作」**，与 **「sched_ext + map」** 区分 **粒度与可重复性**。

### 1.3 执行层在闭环中的位置（与下文步骤对应）

```
S3 输出（结构化调度指令）
        ├─→ 路径一：编码 → 写 BPF map →（可选）sched_ext BPF 读 map → 内核路径
        └─→ 路径二：同一编码 → 用户态调用 syscall / taskset → 线程/进程
```

- **§2 表格中 S4a** 对应「路径一」的 **可达成部分**（map 必做；sched_ext **视 S1**）。  
- **§2 表格中 S4b** 对应「路径二」**全量**。  
- **推荐：** 论文中 **固定一节「执行层并列设计」**，与本节 **1.1 / 1.2** 对齐。

| 步骤 | 具体操作（可写进任务书） | 可行性说明 | 成功判据 | 若未通过时的执行分支 |

| **S5 评估收口** | 固定 workload；对比 **默认 / S4a / S4b / 无预测**；记 **CPU、功耗、延迟、推理开销**。 | **高。** | 表格与图可复现；结论与 **S1～S4 实际走通路径** 一致。 | 减少实验因子；以「本机可达成路径」为主结论，未走通项列入限制。 |

## 3. 路径一：如何验证可行、可行后如何操作

### 3.1 验证操作（按顺序执行，前一步失败则记录原因再决定是否做下一步）

下列 **P1-1～P1-3** 可先在同一终端依次执行（命令与 **§5** 重叠部分保存一份即可）：

```text
adb shell su -c 'zcat /proc/config.gz 2>/dev/null | grep -E "CONFIG_BPF_SYSCALL|CONFIG_DEBUG_INFO_BTF|CONFIG_SCHED_CLASS_EXT|CONFIG_BPF_JIT" | sort -u'
adb shell su -c 'ls -la /sys/fs/bpf; mount | grep bpf'
adb shell getenforce
adb shell su -c setenforce 0
```

（**`setenforce 0`** 仅用于本机实验排查，用完可 **`setenforce 1`** 恢复；论文中须说明。）

| 序号 | 目的 | 具体操作要点 | 判定为「通过」 | 常见失败与含义 |
|------|------|--------------|----------------|----------------|
| P1-1 | 内核是否编入了 BPF / BTF / sched_ext 相关选项 | 见上 **第一条 grep**；若无 config 见 §5 说明 | 至少 **`CONFIG_BPF_SYSCALL=y`** 有记录；**`CONFIG_SCHED_CLASS_EXT`** 为 **y** 或 **n** 有结论 | **无 `/proc/config.gz`**：附录写「无法读取」，结论以 **实测加载能力** 为准 |
| P1-2 | **bpf** 文件系统与挂载 | 见上 **ls + mount** | 存在 **`/sys/fs/bpf`**，root 下列表无 **Permission denied**（或记下受限子路径） | 无目录：路径一 **钉扎/加载** 困难 |
| P1-3 | **SELinux** 与 bpf | **getenforce**；必要时 **setenforce 0** 后 **重复 P1-2**、P1-4 | 记录 **Enforcing/Permissive** 下差异 | **Enforcing** 常阻 **`prog_load`**：需 **sepolicy** 或改 **§4** |
| P1-4 | **最小 BPF 加载**（不要求 sched_ext） | PC **交叉编译** `.o` → **`adb push`** 至 **`/data/local/tmp/`** → **`adb shell su -c`** 用 **静态 `bpftool`** 或 **自写 loader**（`bpf()`）执行 **`prog load` + `attach`**（tracepoint）；子命令以 **bpftool 帮助** 与 **对象类型** 为准 | **`dmesg`** 无 **verifier** 致命错误；**`bpftool prog list`** 或 pin 路径可见对象 | **`EINVAL/EACCES`**：查 **`dmesg`**；调度类程序另受 **SCHED_CLASS_EXT** 限制 |
| P1-5 | **sched_ext 运行时** | 仅当 P1-1 中 **`CONFIG_SCHED_CLASS_EXT=y`**：按 **sched_ext 文档** 用 **`bpftool sched`** 等加载 **scx** 示例（版本与内核对齐）；无工具则附录记 **「配置开、工具未备」** | 能注册 **一种 scx**；**debugfs/sysfs** 下可见 **sched_ext** 状态（以本内核为准） | **多数零售机 n 或工具不齐** → 路径一仅 **观测+map**，落地走 **§4** |

**小结：**  
- **路径一「部分可行」**：P1-4 通过 → 至少可做 **BPF map + tracepoint/kprobe 观测**，并 **向 map 写入**（若 map 与 prog 同时加载成功）。  
- **路径一「含 sched_ext 完整可行」**：P1-5 通过 → 才写 **「BPF 调度类读 map 影响调度」**；否则勿夸大。

### 3.2 判定为可行后的落地操作（建议顺序）

**A. 仅「map + 观测 BPF」（不要求 sched_ext，对应 P1-4 已通）**

1. **在 PC 上**：用 **clang**（目标与本机 **内核架构** 一致，如 **aarch64**）+ **libbpf** 或 **Android/Kernel 提供的头文件**，将 §1.1 中 **map** 与 **tracepoint 程序** 编进 **同一 `.o`**，或 **map + 独立 loader** 按 **libbpf** 惯例 **skeleton** 加载。  
2. **`adb push`** `.o` 与（静态）**bpftool/loader** 到 **`/data/local/tmp/`**（路径可自定，避免 **无执行权限** 的 sdcard）。  
3. **`adb shell su -c`**：  
   - `chmod +x /data/local/tmp/loader`  
   - 运行 loader：**创建 map** → **加载 prog** → **attach tracepoint**（如 `sched/sched_switch`）→ 将 **map fd** 钉到 **`/sys/fs/bpf/`**（若策略允许）或 **仅进程内保留 fd**。  
4. **用户态写策略**：你的 **守护进程/脚本** 根据 S3 输出调用 **`bpf_map_update_elem`**（可用 **C/C++/Rust** 写小工具，或 **libbpf** 生成的 **skeleton** 的 `*_update` 封装），向 **`tid_policy`** 写入 **`sched_hint`**。  
5. **内核侧**：tracepoint 程序中 **`bpf_map_lookup_elem(&tid, ...)`** 仅作 **统计/旁路**（若未加载 sched_ext，**不得** 声称改变 CFS 决策）。  
6. **回滚**：停止写 map、**清空 map** 或 **卸载 prog**（`bpftool prog detach` / 杀 loader）；记录 **`dmesg`**。

**B. 「sched_ext + map」同时可行（对应 P1-5 已通）**

1. 在 **与内核匹配的 scx 源码树**（如 **sched-ext/scx** 或 **kernel/tools/sched_ext**）中，编译目标 **`sched_ext` 类型** 的 BPF 调度器，在 **`select_cpu` / `enqueue`** 等回调中 **`bpf_map_lookup_elem` 读取 `tid_policy`**。  
2. 使用 **`bpftool sched load`**（或当前内核文档中的 **等价命令**）**注册** 该调度器；失败时 **抄录完整命令与 `dmesg`**。  
3. **用户态写 map** 与 **A.4** 相同。  
4. **回滚**：按 **sched_ext 文档** **卸载** 自定义调度器（通常恢复默认 **CFS**）；务必在论文中写清 **命令与风险**。

**C. PC 侧依赖（需在任务书中写清）**

- **NDK** 或 **独立交叉工具链**（**aarch64-linux-android**）；**libbpf** 静态或随 loader 一并推送。  
- **`bpftool`**：建议 **自行编译静态 aarch64 版本** 推送到手机，避免系统镜像无该命令。

## 4. 路径二：如何验证可行、可行后如何操作

### 4.1 验证操作

可先执行（**P2-1～P2-2**）：

```text
adb shell su -c id
adb shell su -c "command -v taskset"
adb shell su -c "taskset --help 2>&1 | head -n 5"
```

**P2-3（建议交互一次，避免引号被 adb 吃掉）：** 顺序执行 **`adb shell`** → **`su`** → **`taskset -p 0x3 $$`**（将当前 shell 绑到 CPU0）→ **`taskset -p $$`** 查看输出掩码；再 **`taskset -p 0xff $$`**（或本机 CPU 全掩码）恢复。返回 **0** 且掩码 **随命令变化** 即通过。

| 序号 | 目的 | 具体操作要点 | 判定为「通过」 | 说明 |
|------|------|--------------|----------------|------|
| P2-1 | root 是否可用 | 见上 **`id`** | 显示 **`uid=0`** | Magisk 弹窗需点允许 |
| P2-2 | 是否有 **`taskset`** | 见上 **`command -v` + help** | 能输出 **用法** | 无则试 **`busybox taskset`** 或 **自写 sched_setaffinity** |
| P2-3 | 能否改 **本 shell** 亲和性 | 见上文 **交互步骤**（`adb shell` → `su` → `taskset -p … $$`） | **`taskset -p $$`** 显示的 **掩码** 与设置一致 | 确认 **亲和性** 未被全局禁用 |
| P2-4 | 能否改 **指定 App 线程** | 启动目标 App，**`adb shell su -c "pidof <包名对应进程名>"`** 得 **PID**；**`ls /proc/$PID/task`** 得 **tid 列表**；对 **某一 tid** 执行 **`taskset -ap 0x0f $tid`**（掩码按 CPU 数改，如 Pixel 6a 上 **先 `grep -c processor /proc/cpuinfo`**） | **无 Operation not permitted**；若失败，试 **同一 UID** 下进程或 **仅系统进程** 以定位 **权限/SELinux** | **部分线程** 可能被 runtime **立刻改回** —— 属 **路径二固有风险**，论文中如实写 |
| P2-5 | （可选）**优先级** | `adb shell su -c "chrt -p <pid> 2>&1"` 或 **`renice`** | 能改 **本机策略允许** 范围内的 **nice/RT** | 失败不否定 **亲和性** 路径，单独记录 |

**小结：** P2-3、P2-4 通过 → **路径二** 在本机上 **可用于闭环实验**（可能仅对部分 tid 有效）。

### 4.2 判定为可行后的落地操作（与 S3 输出联动）

1. **固定目标**：选定 **包名** 或 **进程名**，在论文中写死；启动后 **轮询或一次性** 获取 **PID**（`pidof` / `ps -A | grep`）。  
2. **枚举线程**：`for t in /proc/$PID/task/*; do echo $t; cat $t/comm; done` 或 **`ls /proc/$PID/task`**，将 **tid** 与 **线程名** 对应，便于与 **S3 输出字段**（tid）对齐。  
3. **应用亲和性**（示例：绑定到 **CPU 0–3**，掩码 **0xf**，请按 **`nproc`** 调整）：  
   - 单线程：  
     `adb shell su -c "taskset -ap 0xf <tid>"`  
   - 批量：脚本读取 **JSON/每行 tid+mask**，循环 **`taskset -ap`**。  
4. **与 S3 衔接**：将模型输出的 **`preferred_cpu` 或 affinity mask** 转为 **十六进制掩码**（每一位对应 CPU）；在 PC 上用 **Python** 调 **`adb exec-out su -c 'taskset ...'`** 实现 **自动化闭环**。  
5. **验证是否生效**：  
   - **`taskset -p <tid>`** 查看当前掩码；  
   - 并行录 **Perfetto**「sched slice」，看 **线程是否在预期 CPU** 上运行（**非 100%** 属正常）。  
6. **回滚**：将掩码设回 **`0`~`nproc` 全 1`**，或 **结束进程**；避免长时间把 **系统关键线程** 绑到单核。

## 5. 路径一与路径二均不可用时的备选策略（保险路径）

**有可能。** 在 **固定型号** 真机上，若 **路径一**（`bpf`/`prog_load`/map 或 **sched_ext** 全不可用）且 **路径二**（root 下 **`taskset`/`sched_setaffinity` 仍无法稳定作用**于目标线程，或 **持续被系统改回**）同时成立，则 **无法** 在「本机」上完成原设计的 **内核级/全局用户态执行闭环**。此时应采用下列 **保险路径**（可 **并行** 采用多条），格式与 **§3 / §4** 相同：**适用情形 → 验证操作 → 可行后的落地**。

### 5.1 路径三：纯观测 + 离线策略评估（不依赖本机执行调度指令）

**定位：** **不** 把「预测调度」落实为 **本机实时改调度**，改为：用 **高质量轨迹** 证明 **预测与「若执行则收益」的离线指标**；论文表述为 **策略评估 / 反事实分析** 或 **可观测性验证**。

| 序号 | 目的 | 具体操作要点 | 判定为「通过」 | 说明 |
|------|------|--------------|----------------|------|
| P3-1 | 能否稳定采集 **系统级轨迹** | **`adb`** 下使用 **Perfetto**（`perfetto` CLI 或 **网页配置** 生成 config）；或 **`adb shell atrace`** / **`atrace`** 包一层；至少录 **sched、freq** 相关数据源 | 能重复生成 **.pftrace** 或 **systrace** 文本，时长与场景可控 | 不依赖 root；**路径一/二失败时仍通常可用** |

| P3-3 | PC 侧能否 **重放/对齐** | 将 trace 与 **模型输出时间窗** 对齐（脚本按时间戳 join） | 得到 **「某窗口内若采用策略 S」** 的 **可计算代理指标**（如切换次数、排队时间估计） | **非因果强保证**，须在论文 **局限** 中说明 |
| P3-4 | **论文层面**：**默认调度基线 + 文献/理论上界** | 从 **同一批 trace** 提取 **未注入策略时** 的可重复指标（帧间隔、runqueue 代理、切换率等）作为 **Baseline**；在 **Related Work** 中选 **与调度/资源分配相关的公开结果**（如 **数据中心学习式调度**、**EAS 文档** 所述机制与典型收益叙述等）作 **对照上界** | 正文可写清两段：**（1）本机默认基线数值；（2）文献在其它载体上的改进区间或数量级**，并 **显式声明非同机、非因果执行** | **不替代** P3-1～P3-3 的 **本机数据**；用于 **论证完整性与预期边界**，避免「无执行即无话可写」 |

#### 可行后的落地操作

1. **固定 workload** 与 **录制脚本**（同一场景多次录）。  
2. **S3** 仍输出 **策略序列**，但 **仅写入日志/CSV**，**不** 调用 bpf/`taskset`。  
3. **离线**：用 trace 计算 **默认行为下的指标**；再用 **仿真或启发式** 估计 **若按策略 S 分配 CPU** 的 **上下界**（或仅做 **定性对比**）。

5. **论文**：标题/摘要中避免 **「闭环控制」** 除非补充 **其他载体**（§5.3～5.5）证据。

## 九、阶段性结论（是否支撑课题）

1. **USB 调试与 adb 授权**已通过，`adb devices` 为 **`device`**。  
2. **Magisk root** 已验证（`uid=0`，`magisk` 上下文）。  
3. **内核为 6.1 + Android 14 语境**，具备 **BTF（`/sys/kernel/btf/vmlinux`）** 与基础 **BPF** 配置，**从设备能力角度可支撑 eBPF 探针类课题**。  
4. **未预装 `bpftool`** 属正常现象；课题实施应在 **PC（WSL）交叉编译 arm64 loader + bpf 对象** 后推送运行。  
5. 已完成 **`sched_switch` 实时观测验证**（tracefs 路径）；说明调度事件链路可用。  
6. **主机侧**已完成 `libbpf-bootstrap` 的 **`minimal` 构建验证**（x86-64），证明 BPF 对象与 skeleton 工具链在开发机可闭环。  
7. **真机侧**「运行 arm64 用户态 loader + 自定义 eBPF ringbuf」作为**下一阶段工程验收项**；当前报告以 **T1～T8** 作为已达成阶段性证据，**T9 未完成**如实记录。
